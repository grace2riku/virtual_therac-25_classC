## 1. 目的と適用範囲

本書は、IEC 62304 箇条 6(ソフトウェア保守プロセス)に基づき、リリース後の医療機器ソフトウェアに対する保守活動の計画を定義する。市販後のフィードバック収集から、問題分析、修正実装、再リリースまでのライフサイクルを対象とする。

## 3. ソフトウェア保守計画の確立(箇条 6.1)

### 3.1 保守の目的
- リリース済みソフトウェアに対する問題報告・変更要求を体系的に処理する
- 修正版ソフトウェアを、初回リリースと同等の品質・安全水準で提供する
- リスクマネジメントファイルを継続的に最新に保つ

### 3.2 保守プロセスの構成
保守プロセスは以下の活動で構成される。

| 活動 | 参照条項 | 責任者 |
|------|---------|-------|
| 問題・修正の分析 | 6.2 | {{氏名}} |
| 修正の実施 | 6.3 | {{氏名}} |
| 市販後監視との連携 | ISO 13485 | {{氏名}} |
| リスクマネジメント再評価 | 7.4、ISO 14971 | {{氏名}} |

### 3.3 開発プロセスと保守プロセスの関係
- 保守活動で新規開発・大規模改修を行う場合は、該当部分について箇条 5 のプロセスを再適用する。
- 軽微な修正(バグ修正等)も、箇条 8(構成管理)と箇条 9(問題解決)のプロセスに従って管理する。

## 4. 問題及び修正の分析(箇条 6.2)

### 4.1 フィードバックの監視(箇条 6.2.1.1)

市販後のフィードバックを以下のチャネルで継続的に監視する。

| チャネル | 方法 | 頻度 | 責任者 |
|---------|------|------|-------|
| 顧客問い合わせ | サポート窓口 | 随時 | {{}} |
| 不具合報告 | フィールドサービス | 随時 | {{}} |
| 規制当局の情報 | 薬機法報告、FDA MAUDE 等 | 月次 | {{}} |
| SOUP 供給元情報 | ベンダ公開情報、CVE | 月次 | {{}} |

### 4.2 フィードバックの文書化及び評価(箇条 6.2.1.2)

すべてのフィードバックは課題追跡システムに登録し、以下の観点で評価する。

- 再現性
- 影響を受けるソフトウェアバージョン・構成
- 患者・使用者・第三者への安全性影響
- 既知の残留異常との関連性

### 4.3 問題報告の安全性影響評価(箇条 6.2.1.3)

各問題報告は、リスクマネジメントプロセス(箇条 7.4)へ入力し、新たなハザード又は既存ハザードの再評価要否を判断する。

| 判定 | 処置 |
|------|------|
| 安全性影響あり | リスクマネジメントファイル更新、規制当局報告の要否判断 |
| 安全性影響なし | 通常の問題解決プロセスで処理 |

### 4.4 ソフトウェア問題解決プロセスの使用(箇条 6.2.2)

問題報告の処理は、箇条 9(ソフトウェア問題解決プロセス)に従う。

### 4.5 変更要求の分析(箇条 6.2.3)

変更要求は以下の観点で分析する。

- [ ] 変更内容と目的
- [ ] 影響を受けるソフトウェア項目・ユニット
- [ ] 関連する SRS、設計、試験、リスクコントロール手段
- [ ] 実施に要する工数・期間
- [ ] 患者・使用者への影響及び市販中止・回収の要否

### 4.6 変更要求の承認(箇条 6.2.4)

変更要求は CCB(Change Control Board)で審議し、承認された要求のみ実施に進める。

| 変更区分 | 承認者 |
|---------|-------|
| 軽微(文書のみ、コメント修正等) | {{氏名}} |
| 中程度(非安全関連機能の修正) | CCB |
| 重大(安全関連、アーキテクチャ変更) | CCB + 品質責任者 |

### 4.7 ユーザ及び規制当局への伝達(箇条 6.2.5)

変更内容に応じて、以下の通知義務を遵守する。

| 通知対象 | トリガ | 通知手段 |
|---------|-------|---------|
| ユーザ | 安全関連修正・機能変更 | リリースノート、フィールドアクション |
| 規制当局 | 重篤な不具合・回収 | 薬機法報告、FDA MDR、EU MIR |

## 5. 修正の実施(箇条 6.3)

### 5.1 確立されたプロセスによる修正実施(箇条 6.3.1)

承認された変更は、箇条 5 の開発プロセスおよび箇条 8 の構成管理プロセスに従って実装する。修正の規模に応じて適用するプロセスステップを決定する。

| 修正規模 | 適用ステップ |
|---------|-----------|
| 軽微 | 実装 → ユニット試験 → 構成管理 |
| 中程度 | 詳細設計 → 実装 → ユニット・結合試験 → 構成管理 |
| 重大 | 要求分析 → アーキテクチャ → 詳細設計 → 実装 → 全試験 → 構成管理 |

### 5.2 修正済みソフトウェアの再リリース(箇条 6.3.2)

修正版のリリースは箇条 5.8 のプロセスを適用する。少なくとも以下を実施する。

- [ ] 回帰試験(影響範囲に応じたレベル)
- [ ] リスクマネジメントファイル更新
- [ ] ソフトウェアマスタ仕様書の改訂版作成
- [ ] 既知の残留異常一覧の更新
- [ ] 再現可能性の保証(7 章・ビルド環境のアーカイブ)
