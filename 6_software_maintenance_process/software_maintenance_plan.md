> **本プロジェクトの位置づけ(注記)**
>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。実機・商用販売が存在しないため「市販後監視」「規制当局報告」「顧客フィールドアクション」等の一部プロセスは運用できないが、本書では実製品プロセスの形式をそのまま記述し、本プロジェクトで実運用するサブセットを各章で明示する。本プロジェクトにおける「リリース」は GitHub Releases でのタグ付け(`vX.Y.Z`)を指す。
>
> 題材の **Therac-25** は、初期事故(1985 年 6 月 Kennestone)発生後も AECL の市販後監視・問題解決プロセスが不十分であったため、同型のバグが約 2 年間にわたり 6 件の事故を引き起こした(Leveson & Turner 1993)。「**初期事故の根本原因が特定されていれば後続事故は防げた**」という事実は、保守プロセスの実効性が安全クリティカル機器においてリリース前プロセスと同等に重要であることを示している。本書はこの教訓を反映し、SOUP/コンパイラ更新・SPRP §4.3.1 Therac-25 事故 5 主要因チェックの定期適用・残留異常リストの厳格管理を運用ルールとして組込む。

## 1. 目的と適用範囲

本書は、IEC 62304 箇条 6(ソフトウェア保守プロセス)に基づき、リリース後の Therac-25 Virtual Control Software(TH25-CTRL) に対する保守活動の計画を定義する。保守対象は初回リリース(`v1.0.0` 予定)以降の全バージョンとする。

**保守活動に含まれるもの:**

- リリース後に発見された問題の修正(Corrective maintenance)
- SOUP の脆弱性・不具合公開への対応(Adaptive maintenance)
- 環境変更(コンパイラ / OS / 標準ライブラリ実装 / 依存ライブラリの更新)への追従(Adaptive maintenance)
- 改善要求の反映(Perfective maintenance)
- 新規機能追加(Enhancement — 大規模な場合は開発プロセス箇条 5 を再適用)
- **Therac-25 事故事例から派生する追加学習成果の反映**(プロジェクトの題材性に基づく特殊カテゴリ。例: 公開された他治療装置事故事例の対応 RCM 追加、新規発見された race condition パターンへの対応)

## 3. ソフトウェア保守計画の確立(箇条 6.1)

### 3.1 保守の目的

- リリース済みソフトウェアに対する問題報告・変更要求を体系的に処理する
- 修正版ソフトウェアを、初回リリースと同等の品質・安全水準で提供する
- リスクマネジメントファイル(RMF)を継続的に最新に保つ
- 使用する SOUP の既知不具合・脆弱性を監視し、リスク影響を評価する
- **コンパイラ・標準ライブラリ実装の更新による HZ-007(レガシーコード前提喪失)リスクを継続評価する**(Therac-25 学習目的)
- **新規発見・公開された並行処理パターン関連バグ・整数演算関連バグ・他治療装置事故事例から学習し、対応 RCM を追加する**(Therac-25 学習目的)

### 3.2 保守プロセスの構成

保守プロセスは以下の活動で構成される。すべて単独開発のため作成者が兼任する。

| 活動 | 参照条項 | 責任者 |
|------|---------|-------|
| フィードバック・SOUP 情報の監視 | 6.2.1.1 | k-abe |
| 問題・修正の分析(安全性影響評価含む) | 6.2 | k-abe |
| 変更要求の承認 | 6.2.4 + SCMP §4.1 | k-abe(セルフ、CCB プロセス準拠) |
| 修正の実施 | 6.3 + 箇条 5/8 | k-abe |
| リリース | 6.3.2 + 箇条 5.8 | k-abe |
| 市販後監視との連携 | ISO 13485 | 運用なし(仮想) |
| リスクマネジメント再評価 | 7.4、ISO 14971 | k-abe(RMF 更新) |
| **Therac-25 事故主要因チェックリスト定期適用** | 本書 §4.8(新規) | k-abe |

### 3.3 開発プロセスと保守プロセスの関係

- **軽微・中程度の修正**(バグ修正、非機能変更、SOUP バージョン追従): SCMP §4(構成管理)と SPRP(問題解決)のプロセスに従って管理する。該当箇所のみ箇条 5 の試験を再実施する。
- **重大な変更・新機能追加**: 該当機能に対して箇条 5(開発プロセス)を再適用する。本プロジェクトでは新規インクリメント(Inc.N)として扱う。
- **アーキテクチャ変更**: SRMP §7.4.3 の「新規ハザード出現」相当として、ハザード識別から RCM 検証までの全工程を再実施する。
- **コンパイラ・標準ライブラリ更新**: SCMP §4.3 の「重大かつコンパイラ・標準ライブラリ更新」区分として、全試験再実施 + Sanitizer 全種再実行 + HZ-007 リスク評価を必須化(Therac-6/20 → Therac-25 移行時の前提喪失教訓を現代の C++ 実装に翻案した対応)。

### 3.4 保守フェーズの開始

本プロジェクトでは **Inc.1〜Inc.4 完了後の初回リリース(`v1.0.0`)** をもって保守フェーズを開始する。それ以前の期間は開発プロセス(箇条 5)が主であるが、インクリメント間の既リリース済みコードに対する修正は本計画に準じて実施する。

## 4. 問題及び修正の分析(箇条 6.2)

### 4.1 フィードバックの監視(箇条 6.2.1.1)

リリース後のフィードバックを以下のチャネルで継続的に監視する。

| チャネル | 方法 | 頻度 | 本プロジェクトでの運用 |
|---------|------|------|---------------------|
| 顧客問い合わせ | サポート窓口 | 随時 | 運用なし(仮想)。**GitHub Issues をセルフフィードバック窓口として流用** |
| 不具合報告 | フィールドサービス | 随時 | 運用なし(仮想)。自身での試用・評価による発見を `problem-report` Issue として起票 |
| 規制当局の情報 | 薬機法報告、FDA MAUDE 等 | 月次 | 運用なし(仮想)。実製品展開時に追加 |
| **他治療装置事故事例の文献監視** | 学術論文・FDA Recall 通知・メーカー公開情報 | 四半期 | **本プロジェクト固有: 公開された放射線治療装置事故事例(IEC 60601-2-1 関連リコール、IAEA 報告書等)を四半期で確認し、Therac-25 と類似する事故主要因が新規に判明した場合は SPRP §4.3.1 チェックリストを更新** |
| SOUP 供給元情報 | OSV / GitHub Advisory / CVE / NVD | 月次 + CI 実行時 | **本プロジェクトで実運用**。OSV-Scanner / Trivy の定期実行(GitHub Actions の scheduled workflow)で自動監視 |
| **コンパイラ・標準ライブラリ実装情報** | LLVM Security / GCC Security / Microsoft STL リリースノート | 月次 | **本プロジェクト固有: コンパイラ最適化バグ・UB 挙動変化・メモリオーダリング関連バグの公開を監視。発見時は HZ-007 リスク評価を起動** |
| CI 結果 | GitHub Actions の失敗通知 | コミットごと | **本プロジェクトで実運用** |
| Dependabot 等の自動通知 | GitHub Dependabot | 依存更新あり次第 | **本プロジェクトで実運用**(セットアップ予定) |

**本プロジェクトで実運用する監視活動のまとめ:**

- **常時:** CI 失敗 / Dependabot アラート / OSV-Scanner 検出
- **週次:** GitHub Issues の新規起票確認(自己起票含む)
- **月次:** SOUP 脆弱性スキャン + LLVM/GCC Security 確認
- **四半期:** 他治療装置事故事例の文献監視 + 主要 SOUP のリリースノート確認
- **半期:** SPRP §4.3.1 Therac-25 事故主要因チェックリストの全コードベース適用(本書 §4.8)

### 4.2 フィードバックの文書化及び評価(箇条 6.2.1.2)

すべてのフィードバックは GitHub Issue に登録(SPRP §3.2 に準拠)し、以下の観点で評価する。

- 再現性(再現率、再現条件、Sanitizer 有効/無効での差異)
- 影響を受けるソフトウェアバージョン・構成(どのリリースタグ以降か、コンパイラ・標準ライブラリ実装の組合せ)
- 患者・使用者・第三者への安全性影響(仮想機器でも「実機搭載相当」として評価。Therac-25 の場合は照射対象患者・室内医療従事者・後続使用での残存影響を考慮)
- 既知の残留異常との関連性(マスタ仕様書の残留異常リストとの照合)
- 遡及影響(既リリース版すべてに対する影響か、特定バージョン以降か)
- **Therac-25 事故主要因対応**(SPRP §4.3.1 のチェックリスト A〜F のいずれかに該当するか)

### 4.3 問題報告の安全性影響評価(箇条 6.2.1.3)

各問題報告は、SRMP §7.4(ソフトウェア変更のリスクマネジメント)プロセスへ入力し、新規ハザードまたは既存ハザードの再評価要否を判断する。

| 判定 | 処置 |
|------|------|
| 安全性影響あり(HZ-001〜HZ-010 関連) | **RMF 更新必須**。変更区分は原則 **重大**(SCMP §4.1)。実製品展開時は規制当局報告の要否判断(薬機法 81 条の 2、FDA MDR 等) |
| 安全性影響なし | 通常の問題解決プロセス(SPRP)で処理。重大度 Major/Minor/Trivial に応じ SCMP §4.1 の区分適用 |
| 要調査 | 調査期間中は `severity-critical` + `triage-needed` ラベル付与、24h 以内に再判定 |
| **TSan/UBSan 検出** | SPRP §3.4 に従い、表面症状に関わらず初期判定 Critical。並行処理関連は SPRP §4.3.1 A 型チェック必須 |

### 4.4 ソフトウェア問題解決プロセスの使用(箇条 6.2.2)

問題報告の処理は、SPRP-TH25-001(ソフトウェア問題解決手順書)に従う。保守フェーズで発見された問題も同一の PRB-NNNN 体系で管理する。**保守フェーズで発見された Critical 案件には、SPRP §4.3.1 Therac-25 事故 5 主要因チェックリスト適用に加え、過去リリースへの遡及影響評価を必須化** する。

### 4.5 変更要求の分析(箇条 6.2.3)

変更要求は以下の観点で分析し、Issue 本文に記録する。

- [ ] 変更内容と目的(機能追加 / 修正 / 改善 / SOUP 追従 / コンパイラ追従 / 環境追従)
- [ ] 影響を受けるソフトウェア項目(ARCH-xxx / UNIT-xxx)および関連 SRS / RCM
- [ ] 関連する試験(UT / IT / ST)の再実施範囲、Sanitizer 試験(ASan/UBSan/TSan)の再実施要否
- [ ] 実施に要する工数・期間(本プロジェクトでは学習ペースだが見積もりを残す)
- [ ] 安全性影響(ハザード寄与の有無、重大度、発生確率の変化、Therac-25 事故主要因類型の該当性)
- [ ] リリース方針(次マイナーリリース / パッチリリース / 緊急ホットフィックス)
- [ ] 実製品展開時: 患者・使用者への影響、市販中止・回収の要否

### 4.6 変更要求の承認(箇条 6.2.4)

変更要求は CCB プロセス(SCMP §4.1.1)で審議し、承認された要求のみ実施に進める。

| 変更区分 | 承認プロセス | 本プロジェクトでの運用 |
|---------|-------------|---------------------|
| 軽微(文書のみ、コメント、typo) | セルフ承認 | 即時 PR 可 |
| 中程度(非安全関連機能の修正) | CCB プロセス(1 分インターバル + 自己レビューチェックリスト + CI 全 Pass、学習プロジェクト特例、CCB-TH25-001 §5.4 で正式化予定) | k-abe セルフ |
| 重大(安全関連、アーキテクチャ変更、SOUP 追加/更新、コンパイラ更新) | CCB + リスク影響解析(SRMP §7.4)+ RMF 更新 + 並行処理関連時は TSan 全 Pass | k-abe セルフ、全項目必須化 |

### 4.7 ユーザ及び規制当局への伝達(箇条 6.2.5)

変更内容に応じて、以下の通知義務を遵守する。

| 通知対象 | トリガ | 通知手段 | 本プロジェクトでの運用 |
|---------|-------|---------|---------------------|
| ユーザ | 安全関連修正・機能変更 | リリースノート、フィールドアクション | **リリースノート(GitHub Releases)に記載**。フィールドアクションは運用なし |
| 規制当局 | 重篤な不具合・回収 | 薬機法報告、FDA MDR、EU MIR | 運用なし(仮想)。実製品展開時に追加 |
| コミュニティ | 仮想プロジェクトとしての学習価値の共有、Therac-25 事故対応学習成果 | README 更新、リリースノート、UPSTREAM_FEEDBACK | 本プロジェクトで実運用 |

**リリースノートの必須記載事項(本プロジェクト):**

- バージョン番号(SemVer)と Git タグ
- 含まれる PRB-NNNN / CR-NNNN の一覧
- 安全性に影響する変更の強調(専用セクション)— **対応する HZ-NNN と Therac-25 事故主要因類型(SPRP §4.3.1 A〜F)も明記**
- 既知の残留異常(マスタ仕様書から引用)
- SOUP 更新の一覧(ライセンス変化・脆弱性修正の明示)
- コンパイラ・標準ライブラリ更新がある場合はその旨と HZ-007 リスク評価結果
- アップグレード手順(破壊的変更がある場合)

### 4.8 Therac-25 事故主要因チェックリストの定期適用(本プロジェクト固有)

SPRP §4.3.1 Therac-25 事故 5 主要因チェックリスト(A: race condition / B: 整数オーバフロー / C: 共有状態 / D: インターロック欠落 / E: バイパス UI / F: レガシー前提喪失)を、保守フェーズでも以下のタイミングで定期適用する。

| 適用タイミング | 適用範囲 | 目的 |
|--------------|---------|------|
| 各マイナリリース前 | 当該リリースで変更された全モジュール | 新規導入された欠陥の検出 |
| 半期ごと | コードベース全体 | 既存コードへの新規認識欠陥(他治療装置事故事例から学習した新パターン等)の遡及検出 |
| コンパイラ・標準ライブラリ更新時 | 影響範囲全体 | F 類型(レガシー前提喪失)を中心に再評価 |
| 他治療装置事故事例の新規公表時 | 公表事例と類似性のある全モジュール | 他事故からの学習を本プロジェクトに反映 |

適用結果は `9_software_problem_resolution_process/` 配下の傾向分析レポートに集約し、A〜F 類型での件数推移を追跡する。**3 件以上発生した類型については、SRMP §4.2 の対応カテゴリ再評価と RCM 強化を必須化** する(SPRP §8.2 と整合)。

## 5. 修正の実施(箇条 6.3)

### 5.1 確立されたプロセスによる修正実施(箇条 6.3.1)

承認された変更は、箇条 5(開発プロセス)および箇条 8(構成管理プロセス)に従って実装する。修正の規模に応じて適用するプロセスステップを決定する。

| 修正規模 | 必須ステップ |
|---------|-------------|
| 軽微(ドキュメント整形、コメント、非機能文言) | 実装 → レビュー(セルフ)→ 構成管理(PR + マージ) |
| 中程度(非安全関連機能の修正) | 詳細設計更新 → 実装 → ユニット試験(GoogleTest)→ 結合試験(影響範囲)→ 構成管理 |
| 重大(安全関連、アーキテクチャ変更) | 要求分析 → アーキテクチャ/詳細設計更新 → 実装 → 全レベル試験 → リスク再評価(RMF 更新)→ 構成管理 |
| **重大かつ並行処理関連** | 上記 + **TSan 全 Pass + (該当 RCM がモデル検査対象なら)モデル再検査**(SCMP §4.3 と整合) |
| **重大かつコンパイラ・標準ライブラリ更新** | 上記 + **全試験再実施 + Sanitizer 全種再実行 + HZ-007 リスク評価**(SCMP §4.3 と整合) |
| SOUP 更新 | SOUP 評価(SRMP §4.3)→ 影響解析 → 試験(依存箇所のユニット + 結合 + Sanitizer)→ 構成管理。脆弱性修正の場合は原則迅速対応 |

すべての修正は、**修正前・修正後の動作を証拠として残す**(再現試験の Fail→Pass ログを PR に添付、SPRP §10 に従う)。修正試験ケースは **永続的回帰試験スイートに恒久化** する(SPRP §9 に整合)。

### 5.2 修正済みソフトウェアの再リリース(箇条 6.3.2)

修正版のリリースは箇条 5.8(ソフトウェアリリース)のプロセスを適用する。少なくとも以下を実施する。

- [ ] **回帰試験:** 影響範囲に応じたレベルで実施
  - パッチ: 該当機能のユニット + 結合試験、全システム試験サブセット、ASan/UBSan Pass
  - マイナ: 全ユニット + 全結合 + 全システム試験、ASan/UBSan/**TSan** 全 Pass
  - メジャ: 上記 + 非機能試験(性能、ストレス、負荷、長時間連続運転)+ モデル検査再実行
- [ ] **リスクマネジメントファイル(RMF)の更新:** 安全性影響のある変更の場合
- [ ] **ソフトウェアマスタ仕様書(箇条 5.8)の改訂版作成:** 変更差分と新バージョン番号を反映
- [ ] **既知の残留異常一覧の更新:** 修正された異常を「解決済」に、新規発見された異常を追加
- [ ] **再現可能性の保証:** SCMP §5(構成状態記録)のベースラインに基づき、ビルド環境(コンパイラ・バージョン、`vcpkg.json` または `conan.lock`、CMake バージョン、ツールバージョン)をアーカイブ
- [ ] **リリースタグとリリースノートの作成:** GitHub Releases で `vX.Y.Z` タグ + リリースノートを公開
- [ ] **SHA-256 チェックサム:** リリース成果物のハッシュをリリースノートに記載

### 5.3 バージョニング規則

本プロジェクトはセマンティックバージョニング(SemVer)に準拠する。

| バージョンセグメント | インクリメント条件 |
|--------------------|---------------------|
| MAJOR(X.y.z) | 後方互換性のない API 変更、アーキテクチャ変更、意図する使用目的の変更 |
| MINOR(x.Y.z) | 後方互換性のある機能追加、新規インクリメント完了 |
| PATCH(x.y.Z) | 後方互換性のあるバグ修正、SOUP 更新(機能影響なし) |

開発期間中(初回リリース前)は `0.y.z` とし、Inc.N 完了時に `0.N.0` を付与する。全インクリメント完了・最終システム試験 Pass で `1.0.0` をリリースする。

### 5.4 緊急リリース(ホットフィックス)

安全性影響のある重大不具合が発見された場合は、以下の短縮フローで対応する。**Therac-25 事故では初期事故から 2 年間も同型バグが残存し追加事故を引き起こしたことから、ホットフィックスの実効性は本プロジェクトの学習目的の中核である**。

1. PRB-NNNN 起票(ラベル `severity-critical` + `hotfix`)
2. SRMP §7.4 影響解析を即時実施、RMF 更新
3. **SPRP §4.3.1 Therac-25 事故 5 主要因チェックリストを全コードベースに対して適用**(同型バグの水平展開検出 — Therac-25 が反復事故を起こした最大の教訓への直接対応)
4. 最新リリースタグから `hotfix/PRB-NNNN` ブランチを作成
5. 修正・試験・PR マージ(CCB プロセスは遵守、短縮しない)
6. パッチバージョン発行(`vX.Y.Z+1`)
7. リリースノートに安全性修正として明示(対応 HZ-NNN と SPRP §4.3.1 類型を明記)
8. **修正後 30 日間は週次で同型バグ発生監視**(過去事故の遡及検証 + 新規ハザード発生監視)

## 6. 保守バージョンのサポート方針

| 項目 | 方針 |
|------|------|
| サポート対象バージョン | 最新 MAJOR の最新 MINOR のみ。旧 MAJOR は安全性影響のあるもののみ対応 |
| サポート期間 | 実製品展開時に規定。本プロジェクトでは学習期間中無期限 |
| 非推奨(Deprecation)ポリシー | 非推奨機能は少なくとも 1 MINOR バージョンの猶予期間を設け、マイグレーションガイドをリリースノートに記載 |
| エンドオブライフ(EOL) | MAJOR 更新後 6 ヶ月(実製品展開時の目安。本プロジェクトは仮想のため適用外) |

## 7. SOUP・コンパイラ保守

SOUP(C++ 標準ライブラリ実装、外部ライブラリ、コンパイラ、ビルドツール、Sanitizer ランタイム)の保守について、以下を実施する。

### 7.1 定期監視

- **毎コミット時:** OSV-Scanner / Trivy による脆弱性スキャン(CI)
- **週次:** Dependabot による依存更新プルリクエスト
- **月次:** 主要 SOUP のリリースノート確認(手動)+ LLVM/GCC Security 確認
- **四半期:** 長期サポート(LTS)系の更新判断(コンパイラのメジャー版、CMake のメジャー版など)
- **半期:** SPRP §4.3.1 F 類型(レガシーコード前提喪失)の遡及チェック

### 7.2 更新判断

| 更新種別 | 判断基準 | プロセス |
|---------|----------|---------|
| セキュリティ(CVE 公開) | 影響 CVSS ≥ 7.0、または本ソフトウェアが該当機能を利用 | 即時ホットフィックス候補 |
| バグ修正(該当利用箇所あり) | 該当バグが PRB を発生させうる | 次パッチリリースで取込 |
| 機能追加 | 本ソフトウェアで利用予定あり | 次マイナリリースで取込 |
| **コンパイラ最適化バグ** | UB 関連の挙動変化、メモリオーダリング関連、strict aliasing 関連 | **重大区分として SCMP §4.3 重大かつコンパイラ・標準ライブラリ更新フローを適用 + HZ-007 リスク評価** |
| MAJOR 更新(後方互換性破壊) | 利用 API への影響を個別評価 | マイグレーション計画のうえで次 MINOR / MAJOR で取込 |

### 7.3 更新手順

1. SOUP/コンパイラの新バージョンを CIL に登録(SHA-256 含む)
2. 脆弱性/リリースノートの SRMP §4.3 評価を実施、結果を Issue に記録
3. ローカル環境でロックファイル(`vcpkg.json` ベースライン commit / `conan.lock`)を更新、CI で検証
4. 該当機能のユニット・結合試験を重点実施。**コンパイラ更新時は ASan/UBSan/TSan 全種再実行**
5. CR プロセス(SCMP §4.1)で承認
6. マージ・リリース

## 8. 既知の残留異常の管理

各リリースにおいて未修正の既知異常を「残留異常リスト」として管理する。

- **保管先:** ソフトウェアマスタ仕様書(箇条 5.8)の付録として記載
- **記載項目:** PRB-NNNN、概要、重大度、回避策、暫定評価(リスク受容根拠)、**Therac-25 事故主要因類型対応(SPRP §4.3.1 A〜F)**
- **更新タイミング:** 各リリース直前
- **レビュー:** 新規残留異常追加時はリスク影響評価(SRMP §7.4)を経ること
- **Therac-25 学習目的の追加要件:** 残留異常リストに **A〜F 類型に該当する異常がある場合、その存続期間と回避策の有効性を四半期で再評価** する(Therac-25 では未修正のバグが 2 年間放置された教訓への対応)。同型異常が新規 PRB として追加発生した場合は、残留異常の処理優先度を引き上げる

## 9. 変更影響解析のチェックリスト(保守活動用)

保守フェーズでの各変更に対して以下を確認する(SRMP §7.2 を保守用に拡張)。

- [ ] 変更範囲に RCM を実装するユニット(`// RCM-xxx` タグ)が含まれるか
- [ ] 変更が影響する SRS / ARCH / UNIT / UT / IT / ST の全 ID を列挙できたか
- [ ] 変更により新規 SOUP が追加・更新されるか
- [ ] **変更により新規コンパイラ・標準ライブラリ実装の組合せが導入されるか**(該当時は HZ-007 リスク評価)
- [ ] 変更により既存 API の後方互換性が破壊されるか(バージョンインクリメントの判断根拠)
- [ ] 安全性影響ありの場合、RMF 更新エントリを作成したか
- [ ] **SPRP §4.3.1 Therac-25 事故主要因チェックリスト(A〜F)を適用したか**
- [ ] **並行処理関連変更の場合、TSan 全 Pass を確認したか**
- [ ] リリースノートに含めるべき項目が揃っているか
- [ ] ベースラインタグ(`vX.Y.Z`)の作成準備ができているか
