>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。実機・ハードウェアは存在せず、PC 単体で動作する仮想シミュレータを対象とする。Therac-25 の事故報告書(Leveson & Turner 1993)では、AECL がハードウェアインターロックを廃止しソフトウェア単独で安全機能を実現する設計判断をしたことが事故の主要因の一つとされている。**本プロジェクトでも仮想プロジェクトの性質上、ハードウェア側の独立リスクコントロール手段は存在しないものとして扱い、すべての RCM をソフトウェアで実装する。これは Therac-25 当時の設計前提を学習目的で再現する** ものであり、実機展開時は機械的インターロック等のハードウェア RCM 追加を必須とする。

## 1. 目的と適用範囲

本書は、IEC 62304 箇条 7(ソフトウェアリスクマネジメントプロセス)および ISO 14971:2019 に基づき、Therac-25 Virtual Control Software(TH25-CTRL) のリスクマネジメント活動を **計画** する。本書は「プロセスの計画」であり、具体的なハザード・原因・RCM・検証記録の実体は **リスクマネジメントファイル(RMF)** にて管理する。本書と RMF の関係は以下のとおり:

- **SRMP(本書):** プロセスの方針・方法・判断基準・役割・タイミングを定義する。
- **RMF:** 当該プロセスに従って識別された個々のハザード、原因、RCM、検証記録、残留リスク評価を時系列で記録する生きた文書。
- **SSC:** ソフトウェア安全クラスの決定根拠を記録した静的文書。

本プロジェクトの特殊性として、題材である Therac-25 が **既に致死的事故を起こした実機の再構成** であるため、初期ハザードは想定ではなく **公知事故事例** に基づき確定されている(SSC-TH25-001 §5)。本プロセスは「事故を予防し得たプロセスの追体験」を学習目的に含む。

本プロセスの出力は、本プロジェクトにおいては RMF に統合する(実製品ではシステムレベル RMF に統合する想定)。

## 3. リスクマネジメントプロセスの方針

### 3.1 活動の全体フロー

```
[ハザード識別] → [潜在原因識別] → [事象シーケンス文書化] → [RCM 定義]
   → [RCM 実装(SRS→設計→コード)] → [RCM 検証(試験・解析)]
     → [残留リスク評価] → [受容可否判定] → [RMF 更新]
```

本プロジェクトでは、以下の **2 ルートでハザード識別を行う** ことを特徴とする(クラス C 学習目的の追加要件):

1. **トップダウン**: SSC-TH25-001 §5 で確定済みの公知ハザード(HZ-001〜HZ-010)から原因・RCM を導く。
2. **ボトムアップ**: アーキテクチャ・詳細設計時に **Therac-25 事故の根本原因(race condition、整数オーバフロー、共有状態、レガシーコード前提喪失、バイパス可能 UI)** をチェックリストとして当てはめ、新規ハザードを発掘する。

### 3.2 役割と責任

| 役割 | 責任 |
|------|------|
| リスクマネージャ(本プロジェクトでは作成者が兼任) | 本計画の策定・更新、RMF の一貫性維持、残留リスク受容の判断 |
| 開発者(本プロジェクトでは作成者が兼任) | RCM の実装、試験、トレーサビリティ確保 |
| 検証者(本プロジェクトでは作成者が兼任) | 独立性担保のための時間間隔(SCMP-TH25-001(作成予定)で具体値を定義。学習プロジェクト特例で **1 分以上** を暫定運用、姉妹プロジェクト CCB-VIP-001 §5.4 と同等)とチェックリストに基づく独立検証、CI による機械的検証 |

> **Therac-25 事故と独立性の関係:** Leveson & Turner 1993 では、AECL における **単独開発者による安全クリティカルコード実装と独立レビューの不在** が事故要因として明示的に指摘されている。本プロジェクトは単独開発のため役割は一人で兼任するが、(a) CR 起票から PR 作成までの時間インターバル(SCMP で具体値定義予定、暫定 1 分以上、実機プロジェクトでは 24 時間以上を推奨)、(b) 自己レビューチェックリスト、(c) CI による機械的検証(clang-tidy、cppcheck、GoogleTest、ASan/UBSan/**TSan**、llvm-cov、依存脆弱性スキャン)、の 3 点を独立性の代替手段とする。とくに **TSan(ThreadSanitizer)による並行処理検証は、Therac-25 race condition 事故の直接対応策として必須** とする。

### 3.4 リスク受容基準

ISO 14971 に従い、ALARP(As Low As Reasonably Practicable)の原則で判断する。クラス C ソフトウェアの本プロジェクトに適用する基準:

| 残留リスクレベル | 判定 |
|-----------------|------|
| 死亡/重傷につながるハザードで、RCM 未実装または検証で不合格 | **受容不可**(リリース不可) |
| 死亡/重傷につながるハザードで、RCM が少なくとも 1 段(多くは冗長に 2 段以上)実装・検証合格 | 条件付き受容(RMF に根拠・残留リスク評価を明記) |
| 軽度傷害/健康被害なしで、通常の防御的プログラミング・試験網羅で対処 | 受容可 |

**Therac-25 事故対応ハザード(HZ-001〜HZ-009)** については、上記に加えて以下を必達とする:

- **冗長 2 段以上の RCM** を必須とする(設計安全 + 保護手段の組合せ等)。これは、Therac-25 当時の単一防御策(ソフトウェアチェックのみ)が race condition / 整数オーバフローでバイパスされたことへの直接対応である。
- **並行処理関連ハザード(HZ-002, HZ-003)** については、TSan 実行+モデル検査(TLA+ または等価手段)+境界値ユニット試験の3層検証を必須とする。

RCM の優先順位は ISO 14971 の階層に従う:

1. **設計による本質安全**(ハザード原因の排除・構造的不能化)
2. **保護手段**(防御的チェック、冗長化、フェイルセーフ動作、アラーム)
3. **警告・使用情報**(操作者への通知、取扱説明書記載)

本プロジェクトは仮想機器であり、ハードウェア外部 RCM は存在しないと扱うため、すべての RCM はソフトウェアで実装する。

### 4.3 SOUP 公開異常リストの評価(箇条 7.1.3)

本プロジェクトで使用する SOUP は、以下の手順で既知不具合・脆弱性を評価する。

- **識別:** 採用するすべての C++ ライブラリ・ツール(直接依存・推移的依存を含む)を CIL に登録する。標準 C++ ライブラリ実装(libc++ / libstdc++ / Microsoft STL)・コンパイラ(clang++ / g++)・ビルドツール(CMake / Ninja)・パッケージ管理(vcpkg / Conan)も SOUP として扱う。
- **情報源:**
  - CVE / NVD(<https://nvd.nist.gov/>)
  - GitHub Advisory Database(<https://github.com/advisories>)
  - OSV — Open Source Vulnerabilities(<https://osv.dev/>)
  - 各ライブラリの GitHub Releases / SECURITY.md
  - LLVM Project Security(<https://llvm.org/docs/Security.html>)
  - GCC Security(<https://gcc.gnu.org/wiki/Security>)
  - vcpkg の `vcpkg-tool` リポジトリの脆弱性情報、Conan Center の `cve` フィールド
  - GoogleTest / GoogleMock の Issue Tracker
  - 依存関係スキャナ(Trivy、Dependabot、OSV-Scanner)による自動検出(CI 組込予定)
- **頻度:** 採用時・バージョン更新時・リリース前、および少なくとも各インクリメント境界で実施。
- **評価観点:** ソフトウェア機能として使用している箇所を特定し、当該不具合/脆弱性が HZ-001〜HZ-010 のいずれかに寄与するかを判定する。**コンパイラ最適化に関する既知バグ(strict aliasing、未定義動作の挙動変化等)は、Therac-25 事故と類比的にレガシーコード再利用前提喪失(HZ-007)に分類して評価** する。
- **記録:** 評価結果は CIL の SOUP 節および RMF に記録する。

現時点で具体の SOUP は未確定のため、エントリはアーキテクチャ設計・実装開始時に追加する。

### 4.4 潜在原因の文書化(箇条 7.1.4)

識別された潜在原因は以下の項目で RMF に記録する。

- 原因 ID(例: `CAUSE-HZ002-001`)
- 対応するハザード状況 ID(例: `HZ-002`)
- 寄与するソフトウェア項目(ARCH-xxx / UNIT-xxx)
- 潜在原因カテゴリ(§4.2 のカテゴリ)
- 危害の重大度評価(ISO 14971 基準)
- 発生確率評価(可能な範囲で。Therac-25 事故事例に対応するものは「実証済み」と分類)
- 初期リスクレベル
- **Therac-25 実事故との対応有無**(直接 / 部分 / なし)

### 4.5 事象シーケンスの文書化(箇条 7.1.5)

潜在原因からハザード状況・危害に至る事象シーケンスを明示的に文書化する。形式は因果連鎖とする。

**例 1(Therac-25 race condition、Tyler 第 1 例 相当):**

```
原因: 操作者が X 線モードで設定確定後 8 秒以内に電子線モードへ再編集
      (CAUSE-HZ002-001)
 ↓
中間状態: モード変更指令と磁場電流制御コマンド送信のタイミングが不整合
         (UI 入力処理スレッドと制御スレッドの共有変数アクセスが
          メモリバリア無しで競合)
 ↓
ハザード状況: 電子ビーム高出力モードのまま X 線ターゲットが
             ターンテーブル上に挿入されない (HZ-001 + HZ-002)
 ↓
危害: 患者への 100 倍規模の過剰被曝 → 急性放射線障害・死亡
```

**例 2(Therac-25 整数オーバフロー、Yakima 相当):**

```
原因: 共有フラグ Class3 が 8bit unsigned カウンタで管理され、
      256 回ごとの周期的ゼロリセットを安全チェックロジックが
      「合格」と誤解釈 (CAUSE-HZ003-001)
 ↓
中間状態: ターンテーブル位置確認(in-position チェック)が
         オーバフロー直後の 1 サイクルでスキップされる
 ↓
ハザード状況: ターンテーブル非挿入状態でビームオン許可
             (HZ-003)
 ↓
危害: 過剰被曝 → 急性放射線障害・死亡
```

**例 3(暗号的エラーバイパス、East Texas 相当):**

```
原因: ドーズ異常を検知したが、エラーメッセージ "MALFUNCTION 54"
      のみ表示し、操作者の P キー押下で照射継続を許容
      (CAUSE-HZ006-001)
 ↓
中間状態: 操作者が異常を「装置の通常な誤動作」と誤認識し、
         反復的にバイパス操作を実行
 ↓
ハザード状況: 異常状態が認識されないまま反復照射
             (HZ-006)
 ↓
危害: 致死的被曝の反復 → 患者死亡
```

すべての事象シーケンスは RMF に個別に記録する。

## 5. リスクコントロール手段(箇条 7.2)

### 5.1 リスクコントロール手段の定義(箇条 7.2.1)

各潜在原因に対して、ISO 14971 の階層(§3.4)に従って RCM を定義する。

| RCM 種別 | 想定される手段(Therac-25 文脈の例) |
|---------|----------------------------------|
| 設計による本質安全 | (a) `enum class` + 状態機械による不正状態遷移の構造的排除 / (b) `std::atomic` + メモリオーダリングによる共有状態保護 / (c) lock-free queue によるスレッド間メッセージパッシング(共有変数の排除)/ (d) UI 層と安全コアの分離(箇条 5.3.5)による相互干渉の構造的排除 / (e) 強い型付け(`Energy_MeV`、`DoseUnit_Gy` 等の単位を持つ型)による単位混在の静的防止 / (f) 周期カウンタへの `std::atomic_uint64_t` 使用と境界条件の表明 |
| 保護手段 | (a) ターンテーブル位置の冗長確認(センサ値+モデル予測+前回照射時刻)/ (b) ドーズモニタの 2 系比較とサチュレーション検知 / (c) ウォッチドッグタイマによるデッドロック検出 / (d) インターロック検出時の即時ビームオフ(フェイルセーフ)/ (e) 並行処理アサーション(`std::atomic`、`std::shared_mutex` の意図的使用)/ (f) 単位変換時の境界値表明・オーバフロー検出 |
| 警告・使用情報 | (a) 異常時の **明示的・人間可読** エラーメッセージ(例: "TURNTABLE NOT IN X-RAY POSITION (sensor=ELECTRON_MODE, expected=X_RAY_TARGET)")/ (b) バイパス可能なエラーの限定列挙(致死的エラーは原則バイパス禁止)/ (c) 操作ログの完全保存と表示 / (d) 取扱説明書での既知制限事項記載 |

RCM は `RCM-xxx` の形式で識別し、以下の属性で定義する:

- RCM ID
- 対象原因 ID(1 対多可)
- 手段の種別(上記 3 階層)
- 実装先(ARCH-xxx または UNIT-xxx)
- 検証方法(予定)
- 検証試験 ID(予定)
- **Therac-25 事故主要因への対応(該当する場合)**

### 5.2 ソフトウェアで実装されるリスクコントロール手段(箇条 7.2.2)

ソフトウェアで実装する RCM は以下の追跡を徹底する。

1. **SRS への要求事項としての組込み:** RCM の機能要求を SRS に `SRS-xxx` として記述し、本文中に `RCM-xxx` を参照明記する。
2. **アーキテクチャ設計での配置:** SAD において RCM を実装するソフトウェア項目を特定する。クラス C 分離設計(箇条 5.3.5)の対象にもなる。**UI 層と安全コアの分離は Therac-25 の race condition 直接対応として最優先で検討** する。
3. **詳細設計での実装:** SDD においてアルゴリズム・インタフェース・不変条件を記述する。並行処理 RCM については **TLA+ 等によるモデル検査の検討対象** とする。
4. **コードでのタグ付け:** 該当関数/モジュールのドキュメンテーションコメント(Doxygen 等)に `// RCM-xxx` コメントを付与し、grep 可能にする。`std::atomic` / `std::mutex` 使用箇所には対応する RCM ID をコメントで紐付ける。
5. **試験での検証:** ユニット試験(GoogleTest)・結合試験・システム試験で RCM の動作を確認する。試験ケースに `RCM-xxx` をタグ付けする。並行処理 RCM については **TSan 必須実行** とする。

## 6. リスクコントロール手段の検証(箇条 7.3)

### 6.1 リスクコントロール手段の検証(箇条 7.3.1)

各 RCM は以下のいずれかの方法で検証し、記録を残す。

| RCM 種別 | 推奨される検証方法 |
|---------|-------------------|
| 設計による本質安全 | 設計レビュー、静的解析(clang-tidy / cppcheck)、型チェック、(必要に応じて)形式手法(TLA+ / SPIN / Frama-C) |
| 保護手段(防御的ロジック) | ユニット試験(GoogleTest による境界値・異常系試験、libFuzzer によるプロパティ試験) |
| 保護手段(冗長化・フェイルセーフ) | 結合試験(故障注入シナリオ) |
| **保護手段(並行処理・タイミング)** | **ユニット試験(TSan 有効化ビルド)+ 結合試験(タイミング窓を強制注入する試験)+ モデル検査(TLA+ による状態空間網羅) — Therac-25 race condition 対策として 3 層必須** |
| 保護手段(ウォッチドッグ等) | 結合試験/システム試験(遅延注入・故障注入) |
| 警告(エラー表示・操作確定) | システム試験(異常シナリオでの表示確認、IEC 62366-1 ユーザビリティ妥当性確認) |
| メモリ安全(バッファ・ヌルポインタ) | ユニット/結合試験(ASan / UBSan 有効化ビルド) |

記録は該当試験報告書(UT-xxx / IT-xxx / ST-xxx)に残し、RMF からトレーサブルにする。

### 6.2 新たな事象シーケンスの文書化(箇条 7.3.2)

RCM 実装自体が新たなハザード状況を生む可能性を以下の観点で評価し、該当あれば §4.5 の事象シーケンスに追加する。

- [ ] RCM が動作しない場合の挙動(安全側/危険側)
- [ ] RCM 誤動作(誤検出)による想定外停止・警告過多 — **誤検出によるバイパス常態化(HZ-006 と同型)を誘発しないか特に注意**
- [ ] RCM 実装によるリソース消費増で別リソース枯渇を誘発
- [ ] RCM 実装が別の RCM の前提を崩す(副作用)
- [ ] **並行処理 RCM(`std::mutex` 等)導入によるデッドロック・優先度逆転の発生**
- [ ] **冗長化 RCM が逆に共有状態(共通モード故障)を増加させていないか**

評価は各 RCM 実装完了時および変更発生時に実施する。

### 6.3 トレーサビリティの文書化(箇条 7.3.3)

以下のトレーサビリティを RMF および各ドキュメントのトレーサビリティマトリクスで維持する。

```
[ハザード HZ-xxx] ←→ [潜在原因 CAUSE-xxx] ←→ [RCM-xxx]
                                                 ↓
                                         [SRS-xxx] ←→ [ARCH-xxx / UNIT-xxx]
                                                                 ↓
                                                         [UT-xxx / IT-xxx / ST-xxx]
```

ID 体系は CLAUDE.md「ドキュメント間のトレーサビリティ」表および SDP §3 の命名規約に準じ、RMF で一元管理する。

## 7. ソフトウェア変更のリスクマネジメント(箇条 7.4)

### 7.1 変更がソフトウェアに与える影響の解析(箇条 7.4.1)

すべてのソフトウェア変更(要求変更、設計変更、不具合修正、SOUP 更新を含む)について、以下を解析する。

- 新たに発生し得るハザード状況
- 既存ハザード状況の重大度・発生確率の変化
- 新たな潜在原因の出現
- **Therac-25 事故主要因類型(race condition / 整数オーバフロー / レガシーコード前提喪失 / バイパス UI 等)への該当性**

変更は Pull Request 単位で管理し、PR 記述欄に影響解析結果を記載する。クラス C 項目に影響する変更は、影響解析結果を RMF に追記する。

### 7.2 既存 RCM への影響の解析(箇条 7.4.2)

以下のチェックリストで評価する。

- [ ] 変更範囲に RCM を実装するユニット(`// RCM-xxx` タグで `grep` 検出可能)が含まれるか
- [ ] RCM が依存するインタフェース・データ構造・定数に変更があるか
- [ ] RCM の前提(例: センサ値のレンジ、タイマ精度、メモリオーダリング)が変わるか
- [ ] 関連する検証記録(UT-xxx / IT-xxx / ST-xxx)は再実施が必要か
- [ ] 変更により新規 SOUP が追加・更新されるか(追加時は §4.3 を実施)
- [ ] **並行処理関連の変更か(`std::atomic`、`std::mutex`、`std::thread` 関連、共有変数追加・削除)。該当時は TSan 有効化ビルドでの再試験を必須とする**
- [ ] **コンパイラ・C++ 標準ライブラリ実装の変更を伴うか(該当時はレガシーコード前提喪失 HZ-007 リスクを評価)**

### 7.3 解析結果に基づくリスクマネジメント活動(箇条 7.4.3)

解析結果に応じて、下記の活動を選択的に再実施する。記録は RMF に残す。

| 影響の程度 | 必要活動 |
|-----------|---------|
| 新規ハザード出現 | ハザード識別 → 原因識別 → RCM 定義 → 実装 → 検証(全工程再実施) |
| 既存 RCM の論理変更 | 当該 RCM の再設計・再検証 |
| 既存 RCM の依存変更のみ(論理不変) | 当該 RCM の再検証(試験再実施) |
| **並行処理関連の変更** | **TSan 再実行 + 関連結合試験再実施 + (該当 RCM がモデル検査対象なら)モデル再検査** |
| RCM 非関連部の変更 | リグレッション試験範囲の判定のみ |
| SOUP 更新 | §4.3 の SOUP 評価を再実施 |
| コンパイラ・標準ライブラリ更新 | 全試験再実施 + Sanitizer 全種再実行 + HZ-007(レガシーコード前提喪失)リスク評価 |
