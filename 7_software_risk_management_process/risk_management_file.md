> **本プロジェクトの位置づけ(注記)**
>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。本 RMF は ISO 14971:2019 に基づく機器全体のリスクマネジメントファイルの雛形であり、ソフトウェア固有の活動計画は [`software_risk_management_plan.md`](./software_risk_management_plan.md)(SRMP-TH25-001)を参照する。
>
> **本 RMF は「生きた文書」である** — ハザードの新規識別、RCM 定義・検証、残留リスク再評価、市販後情報の反映のすべてを本書に追記し続ける。更新ごとにバージョンを改訂し、改訂履歴に変更理由を記録する。
>
> **本プロジェクト固有の重要事項:** 題材の **Therac-25** は 1985-1987 年に米国・カナダで 6 件の放射線過剰照射事故(6 名重症または死亡)を起こした実在の医療機器である(AECL、1982 年発売、Leveson & Turner 1993)。HZ-001〜HZ-010 のうち HZ-001/HZ-002/HZ-003/HZ-004/HZ-006/HZ-007 は **当時 P5(頻繁)〜P4(可能性大)に相当する頻度で実際に発現した** ことが歴史的に確認されている。本 RMF の発生可能性推定は、この **公知の実証事例** に基づき、想定値ではなく実績値として記録する点が他の医療機器プロジェクトと根本的に異なる。

- リスクマネジメントレビュー(箇条 9)— §9 に記録

## 3. 意図する使用及び安全に関連する特性(ISO 14971 箇条 5.2)

### 3.1 意図する使用

| 項目 | 記述 |
|------|------|
| 対象疾患・適応 | 悪性腫瘍に対する放射線治療(電子線治療および光子線/X 線治療)。腫瘍細胞を高エネルギー電離放射線で破壊し、周囲正常組織への損傷を最小化する |
| 対象ユーザ | 放射線治療医、医学物理士、放射線治療技師。**処方は治療医、線量計算・装置設定は医学物理士、装置操作は治療技師が分担担当する想定。操作者は医療機関での集中訓練を完了済の有資格者** |
| 使用環境 | 放射線治療室(遮蔽壁・遮蔽扉付の専用施設)。操作者は別室の制御コンソールから装置を遠隔操作 |
| 操作頻度 | 1 日複数患者(典型的に 30〜50 名/日)。1 患者あたり数分〜十数分の照射、複数日連続(分割照射)で完了 |
| 接触対象 | 患者の体表・体内に高エネルギー電離放射線を直接照射(電子線: 5-25 MeV、X 線: 6-25 MV) |
| 患者集団 | 主に成人がん患者。小児・高齢者・脆弱患者を含む |

> 本プロジェクトは **PC 単体のシミュレータ** だが、意図する使用は実機(Therac-25 相当機)を想定して記述する(安全クラス C 判定の前提、SSC-TH25-001 §3)。実機展開時は本節を再検証する。

### 3.2 安全に関連する特性(ISO 14971 附属書 C)

| 項目 | 記述 |
|------|------|
| エネルギーを与えるか | **はい — 高エネルギー電離放射線(電子線 5-25 MeV、制動 X 線 6-25 MV)を患者に直接照射する**。**Therac-25 事故の最大リスク源**。誤った高出力モードでのターゲット非挿入は処方量の 100 倍規模の被曝を発生させ得る |
| 物質を投与するか | いいえ |
| 生体由来物質との接触 | いいえ(非接触遠隔照射) |
| 滅菌の要否 | いいえ |
| 継続使用時間 | 連続(数分〜十数分/患者)。1 日複数患者で連続稼働。週次・年次の定期保守時はビームオフ |
| 操作ミス可能性 | **高い** — 治療計画の設定変更時に複数の入力(モード、エネルギー、コリメータ開口、ターンテーブル位置、照射時間、ドーズ単位)を順次入力し、最終確定後にビームオン。**Therac-25 事故では 8 秒以内の編集で race condition が発現** したことが Tyler 事故第 1 例で確認されている |
| ソフトウェア制御 | **あり(クラス C)** — ビームモード制御、エネルギー設定、磁場制御、ターンテーブル位置決め、コリメータ制御、ドーズモニタ、インターロック監視、操作者 UI、ログがすべてソフトウェアで実現。**Therac-25 ではハードウェアインターロック(Therac-6/20 までは機械式連動)を廃止しソフトウェア単独で安全機能を実現した設計判断が事故主要因の一つ**(SSC-TH25-001 HZ-007) |
| 外部インタフェース | 治療計画システム(TPS)からの処方データ入力(将来拡張)。本版は単独操作 |
| 測定機能 | ドーズモニタ(2 組のイオンチェンバ)、ビーム電流、ターンテーブル位置センサ、温度・冷却水・真空度センサ、ドア開閉インターロック |
| 一回使用/再使用 | 装置本体は多数患者に再使用、患者位置決め用補助具は患者ごとに調整 |
| 並行処理 | **あり** — 操作者 UI 入力処理、ビーム制御ループ、ドーズモニタ読取、インターロック監視は **並行タスク** として動作。**Therac-25 race condition 事故(Tyler 第 1 例)はこの並行性に起因** |

## 4. ハザード・ハザード状況・危害の識別(ISO 14971 箇条 5.4)

### 4.1 ハザード一覧

SSC-TH25-001 で識別した 10 件を初期ハザードとして登録する。以降、SRS/SAD/実装/試験/市販後情報の各段階で追加・更新する。**HZ-001/HZ-002/HZ-003/HZ-004/HZ-006/HZ-007 は Therac-25 事故により実際に発現した実証ハザードである。**

| ハザード ID | ISO 14971 分類 | 記述 | 想定される危害 | ソフトウェア寄与 | Therac-25 実事故との対応 | 発生しうる機序(概要) |
|-----------|----------------|------|------------|--------------|---------------------|-------------------|
| HZ-001 | 機能的(放射線照射) | **電子ビーム高出力モードのまま X 線用ターゲット・フラットニングフィルタが挿入されない状態で患者に照射**(処方量の 100 倍規模の過剰被曝) | 急性放射線障害・皮膚潰瘍・組織壊死・死亡 | **有** | **直接対応**(Tyler/Hamilton/East Texas 等、全事故の中核原因) | モード制御ロジック誤り、ターンテーブル位置確認のバイパス、ビームオン許可ロジック誤り |
| HZ-002 | 機能的(並行処理) | 操作者入力処理と磁場/ターンテーブル制御間の **タイミング競合(race condition)** により、設定変更が未反映のままビーム照射が実行される | HZ-001 と同等(致死的被曝) | **有** | **直接対応**(Tyler 事故第 1 例: 8 秒以内編集で発現) | UI 入力処理スレッドと制御スレッドの共有変数アクセスがメモリバリア無しで競合 |
| HZ-003 | 機能的(整数演算) | 共有フラグ変数の **周期的ゼロ値(整数オーバフロー)** により、本来行われるべき安全チェック(ターンテーブル in-position 確認)がバイパスされる | HZ-001 と同等(致死的被曝) | **有** | **直接対応**(Yakima 事故: Class3 変数のオーバフローバグ) | unsigned カウンタの周期的リセット、安全チェック分岐条件のカウンタ値依存 |
| HZ-004 | 機能的(モニタリング) | **インターロック監視ロジック**(ドア、ビーム電流、ターンテーブル位置、ドーズモニタ)の検知失敗・実装不備により、危険状態が検知されずビームが継続 | 被曝継続による致命的障害(患者・室内医療従事者) | **有** | **直接対応**(ハードウェアインターロック廃止 → ソフトウェア依存の設計判断) | 単一防御層への依存、防御層の論理矛盾、防御層の競合、ビームオフ指令発行ロジックの不備 |
| HZ-005 | 機能的(計算) | ビーム出力・ビームパルス数の計算/積算誤り(単位変換誤り、丸め誤差、オーバフロー)による過剰/過少被曝 | 過剰被曝による急性放射線障害・死亡、または過少被曝による治療失敗 | **有** | 部分対応(Therac-25 では直接的原因ではなかったが、放射線装置の一般的既知リスク) | 浮動小数誤差、整数オーバフロー、単位変換誤り、桁落ち、終了判定誤り |
| HZ-006 | 情報的・操作的 | 暗号的なエラーメッセージ(例: "MALFUNCTION 54")と **操作者によるバイパス常態化**(P キー押下で照射継続)の UI 設計 | 致命的な異常を異常と認識できず、照射が反復継続される | **有** | **直接対応**(全事故で共通して観察された運用パターン、特に East Texas) | エラーコードのみの表示、操作者の継続操作で復旧、ログ記録なし、致死的エラーのバイパス許容 |
| HZ-007 | 機能的(レガシー) | 旧機種(Therac-6/20)由来の **コード再利用における前提条件見直し漏れ**(旧機種ではハードウェアインターロックが同等機能を担っていた) | ソフトウェア単独では検知できない危険状態の潜伏 | **有** | **直接対応**(AECL 事故後分析の主要指摘事項) | 依存していた外部条件の消失、API セマンティクス変化、過去仕様に依存した実装の見直し漏れ |
| HZ-008 | 機能的(モニタリング) | ドーズモニタ(2 組のイオンチェンバ)の読み取り処理・比較ロジックの誤り(片系失陥時のフォールバック、サチュレーション検知等) | 過剰被曝を検知できずビーム継続 | **有** | 部分対応(閉ループ制御の妥当性に関する一般的リスク) | 二系統比較ロジック誤り、サチュレーション検知漏れ、片系失陥時のフォールバック誤動作 |
| HZ-009 | 機能的(状態管理) | 電源喪失・再起動時の不定動作(治療中断状態からの誤った再開、ビームオン状態の引き継ぎ) | 意図しない二重被曝・過剰被曝 | **有** | 部分対応 | 永続化破損、起動時復元誤り、ビームオン状態管理の不備、再起動フロー未検証パス |
| HZ-010 | 情報的・サイバーセキュリティ | 操作者ログイン・権限管理の不備による、未承認の処方変更・ビーム設定変更 | 不適切な治療条件での照射 | **有** | 潜在リスク(Therac-25 当時は未認識、現代ではサイバーセキュリティ観点で必須) | 認証バイパス、権限昇格、操作者識別欠如、設定変更ログ欠如 |

> ソフトウェア寄与ありのハザードは SRMP-TH25-001 §4.1 の「ハザード寄与解析」と 1 対 1 に対応する。ID が一致するように維持する。

### 4.2 事象シーケンス(代表例 — Therac-25 実事故ベース)

各ハザードに対する代表的な事象シーケンスを以下に示す。**Therac-25 の事象シーケンスは公知事故報告書(Leveson & Turner 1993)に基づく実例であり、想定ではなく実証** である。事象 ID は `EV-HZxxx-nnn` 形式で管理する。

#### EV-HZ001-001 / EV-HZ002-001(Tyler 第 1 例 race condition、1985 年 6 月)

```
[原因] 操作者が X 線モードで処方確定後、8 秒以内に電子線モードへ再編集
        (CAUSE-HZ002-001)
  ↓
[中間状態 1] UI 入力処理スレッドが新モード(電子線)を内部状態に書込み
  ↓
[中間状態 2] 制御スレッドが旧モード(X 線)用の磁場電流コマンドを電源に発行済み
        (タイミング窓: 8 秒以内 = 制御スレッドの再評価周期内)
  ↓
[中間状態 3] ターンテーブルが X 線用ターゲット位置にあるべきだが、
        新モード(電子線)に対応する位置(散乱箔)に切り替わってもいない不整合状態
        — 共有変数アクセスがメモリバリア無しで race condition 発生
  ↓
[ハザード状況] 電子ビーム高出力モード(X 線生成用の高エネルギー)のまま
        X 線ターゲットが ターンテーブル上に挿入されない (HZ-001 + HZ-002)
  ↓
[危害] 患者への 100 倍規模の過剰被曝(推定 1.5 万 rad 相当)
        → 急性放射線障害 → 死亡
```

**実発生事例:** Tyler 第 1 例(1985 年 6 月、Kennestone Regional Oncology Center)、Tyler 第 2 例(1986 年、East Texas Cancer Center)、Hamilton(1985 年 7 月、Ontario Cancer Foundation)等、**少なくとも 4 件** で本シーケンスまたは類似シーケンスが発現。

#### EV-HZ003-001(Yakima Class3 変数オーバフロー、1987 年 1 月)

```
[原因] 共有フラグ Class3(8bit unsigned カウンタ)が周期的に 0 にリセット
        される瞬間に、安全チェック(ターンテーブル in-position 確認)
        ロジックが「合格」と誤解釈
        (CAUSE-HZ003-001、ソースコードレベルでは Class3++ 操作の 256 周期問題)
  ↓
[中間状態 1] ターンテーブル位置センサ読取時に Class3 == 0 が瞬間的に成立
  ↓
[中間状態 2] 安全チェックロジックが Class3 == 0 を「位置確認済」と誤解釈
        (本来 Class3 はインクリメント済の蓄積カウンタとして使用されるべき)
  ↓
[ハザード状況] ターンテーブル非挿入状態でビームオン許可
        (HZ-003 → HZ-001 連鎖)
  ↓
[危害] 過剰被曝 → 急性放射線障害 → 患者死亡
```

**実発生事例:** Yakima Valley Memorial Hospital(1987 年 1 月)、患者死亡。Therac-25 全 6 事故のうち 6 件目で、本バグが Tyler/Hamilton/East Texas 事故とは **別の独立した根本原因** であったことが事後分析で判明した(Leveson & Turner 1993 §IV.G)。

#### EV-HZ006-001(East Texas MALFUNCTION 54 バイパス、1986 年 3-4 月)

```
[原因] 治療制御ソフトウェアがドーズ異常または装置内部異常を検知し
        "MALFUNCTION 54" エラーコードを画面表示するが、
        ビームは停止せず操作者の P キー(Proceed)押下で照射継続を許容
        (CAUSE-HZ006-001、UI 設計上の致死的誤り)
  ↓
[中間状態 1] 操作者が異常を「装置の通常な誤動作」と誤認識
        (頻繁に発生する非致死的エラーと区別されない暗号的メッセージ)
  ↓
[中間状態 2] 操作者が反復的にバイパス操作(P キー押下)を実行
        — ログには「Mode: Field Light, MALFUNCTION 54, P key pressed」
        とのみ記録され、致死的被曝の記録がない
  ↓
[ハザード状況] 異常状態が認識されないまま反復照射 (HZ-006)
  ↓
[危害] 致死的被曝の反復 → 患者死亡
```

**実発生事例:** East Texas Cancer Center(1986 年 3 月および 4 月、同一患者への 2 回連続事故)、患者死亡。**1 回目の事故で操作者が異常を察知できなかったため 2 回目の事故が発生した** ことが、エラー UI 設計の致命的欠陥を象徴する事例として記録されている。

#### EV-HZ004-001(インターロックロジックのデッドロック)

```
[原因] インターロック監視タスクが他タスクとのリソース競合または
        論理矛盾でデッドロック状態に陥る
  ↓
[中間状態 1] ドア開放・ビーム異常・ターンテーブル位置異常等を検知しても
        ビームオフ指令が発行されない
  ↓
[中間状態 2] 装置は「インターロック監視中、異常なし」と認識し続ける
  ↓
[ハザード状況] 危険状態でビームが継続 (HZ-004)
  ↓
[危害] 室内医療従事者・患者の被曝 → 急性放射線障害・死亡
```

**実発生事例:** Therac-25 で直接記録された事例はないが、**ハードウェアインターロックを廃止した設計判断(HZ-007)** がこのリスクを構造的に内包している。Leveson & Turner 1993 では、当時の AECL がインターロックロジックの故障モード解析(FMEA/FTA)を実施していなかったことが指摘されている。

#### EV-HZ007-001(Therac-6/20 コード再利用前提喪失)

```
[原因] AECL は Therac-25 の制御ソフトウェアを Therac-6/20 から
        相当部分を再利用したが、Therac-6/20 では機械式インターロックが
        ソフトウェアの安全チェック失敗を冗長カバーしていた前提を見直さなかった
        (CAUSE-HZ007-001)
  ↓
[中間状態 1] Therac-25 ではハードウェアインターロックを廃止したため、
        ソフトウェア単独の安全チェックが唯一の防御層
  ↓
[中間状態 2] Therac-6/20 由来のソフトウェアバグ(後の race condition / Class3
        オーバフロー)が機械的冗長カバーを失い顕在化
  ↓
[ハザード状況] ソフトウェア単独では検知できない危険状態が潜伏 (HZ-007)
  ↓
[危害] HZ-001/HZ-002/HZ-003 の前提条件として、すべての Therac-25 事故に間接寄与
        → 6 件 6 名の重症・死亡事故
```

**実発生事例:** AECL 事故後分析(1987-1988)で、**Therac-25 の 6 事故すべてに Therac-6/20 コード再利用前提の見直し漏れが間接寄与している** と結論された(Leveson & Turner 1993 §IV.A)。

全事象シーケンスは SRS/SAD 完成に合わせて拡充する。**他治療装置事故事例の四半期文献監視(SMP §4.1)で新規シーケンスを発見した場合も本節に追加する**。

### 6.3 コントロール種別の優先順位(ISO 14971 箇条 7.1)

ISO 14971 の階層に厳密に従う:

1. **設計による本質的安全**(最優先)
   - ハザード原因の構造的排除(例: `enum class` による不正遷移防止、`std::atomic` による共有状態保護、UI 層分離による race condition 不可能化、強い型による単位混在静的防止)
2. **製品自体又は製造工程における保護手段**
   - 冗長チェック、ウォッチドッグ、自己診断、フェイルセーフ停止、アラーム
3. **安全に関する情報**(最低優先)
   - 取扱説明書、画面上の警告、ラベル

本プロジェクトではソフトウェアのみで RCM を実装するため、ハードウェア外部手段は考慮しない(SSC-TH25-001 方針)。**Therac-25 学習目的の最重要点として、Therac-25 当時に欠落していた「設計による本質的安全」(RCM-001/002/003/004/008/010/012/013/017/018/019)を最優先で実装する**。

### 8.2 ベネフィットとリスクの対比

**初期評価(Phase 2 時点):**

ベネフィット:
- 高エネルギー電離放射線による精密ながん治療を可能とし、外科手術が困難な腫瘍位置・形状に対する効果的な治療法を提供する
- 電子線治療(浅い腫瘍向け)と X 線治療(深い腫瘍向け)の両モードを 1 台で提供し、治療施設の機器コストと設置面積を削減する
- 放射線治療は外科手術と並ぶ標準的がん治療法であり、Therac-25 相当機なしでは多くの患者が治療機会を失う

残留リスク(想定):
- 全 RCM が実装・検証された場合、§6.1 の目標どおりであれば残留リスクはすべて「中」以下に収まる見込み
- 実機ではハードウェア側独立 RCM(機械式インターロック・独立ドーズモニタ等)の追加により更に低減可能だが、本プロジェクトは仮想のためソフトウェア内 RCM のみで達成する。**Therac-25 事故の最大教訓「ハードウェアインターロック廃止の不適切な判断」を本プロジェクトは意図的に踏襲し、ソフトウェア単独で安全機能を実現することの困難さを学習する**

**最終判定:**

初回リリース前に §8.1 のチェックリストを充足した段階で本節を更新し、判定記録を残す。

## 10. 製造後の情報収集及びリスク再評価(ISO 14971 箇条 10)

**本プロジェクトは仮想のため実製造後監視は運用しない** が、SMP-TH25-001 §4.1 に従い以下の枠組みを維持する。

### 10.1 情報源

- GitHub Issues(セルフフィードバック窓口)
- SOUP 脆弱性情報(OSV、GitHub Advisory Database、CVE/NVD、LLVM Security、GCC Security)
- Dependabot アラート
- CI 失敗(C++ ビルド、GoogleTest、ASan/UBSan/TSan、llvm-cov、OSV-Scanner、Trivy)
- **他治療装置事故事例の四半期文献監視**(SMP §4.1)— Therac-25 と同型の事故が他装置で再発した場合に学習を本プロジェクトに反映
- **コンパイラ・標準ライブラリ実装情報の月次監視**(SMP §4.1)— LLVM/GCC Security
- (実製品展開時追加)顧客フィードバック、FDA MAUDE、PMDA 医療機器不具合情報、IEC 60601-2-1 関連リコール、IAEA 報告書

### 10.2 収集頻度

- 常時: GitHub Issues、CI 失敗、Dependabot
- コミットごと: OSV-Scanner / Trivy スキャン
- 月次: SOUP/コンパイラリリースノート確認、LLVM/GCC Security 確認
- 四半期: LTS 系更新判断、他治療装置事故事例文献監視
- 半期: SPRP §4.3.1 Therac-25 事故主要因チェックリスト全コードベース適用(SMP §4.8)

### 10.3 再評価のトリガ

以下のいずれかが発生した場合、本 RMF の該当箇所を更新し、バージョンを改訂する:

- 新規ハザードの識別(他治療装置事故事例から学習した新パターン含む)
- 既存ハザードの発生頻度・重大度が推定を超えた
- RCM の無効化・劣化(試験 Fail、実運用での機能不全)
- SOUP の重大脆弱性・不具合の公表(CVSS ≥ 7.0 または利用箇所に影響)
- **コンパイラ・標準ライブラリ実装更新**(HZ-007 リスク評価必須、SCMP §4.3 重大区分)
- ソフトウェア変更(SRMP §7.4 の影響解析で「新規ハザード出現」判定)
- **TSan/UBSan の CI 新規検出**(CCB-TH25-001 §6.2 臨時会トリガと整合)
- **SPRP §4.3.1 で 3 件以上発生類型検出**(SRMP §4.2 再評価必須)

### 10.4 処置

1. 本 RMF 更新(該当節に追記、改訂履歴に記録)
2. 関連ドキュメント(SRS / SAD / SDD / 試験計画・報告)を同一 PR 単位で更新
3. SRMP §7.4 に従いソフトウェア変更のリスクマネジメントを実施
4. 必要に応じて SPRP(PRB)→ SCMP(CR)→ 実装 → 検証 → リリース
5. **Therac-25 事故主要因類型対応の場合**: SPRP §4.3.1 全コードベース水平展開チェック実施(SMP §5.4 ホットフィックス時の手順と整合)
