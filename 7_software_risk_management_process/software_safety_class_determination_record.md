## 4. ソフトウェア安全クラスの定義(IEC 62304 箇条 4.3 より)

| クラス | 定義 |
|--------|------|
| A | 傷害又は健康被害はあり得ない |
| B | 重傷には至らない傷害があり得る |
| C | 死亡又は重傷があり得る |

クラス決定は、**ハードウェア外部リスクコントロール手段を考慮しない前提**で、ソフトウェアが寄与し得るハザード状況から生じ得る危害の重大度に基づく。

## 7. 安全クラスの決定

### 7.1 決定結果

**本ソフトウェアの安全クラス: C**

### 7.2 根拠

以下の観点から、本ソフトウェアはクラス C に該当する。

- [ ] 故障時に死亡又は重傷が生じ得るハザード状況に寄与する({{該当ハザード ID}})
- [ ] ハードウェア外部リスクコントロール手段でソフトウェア故障を完全に緩和できない
- [ ] 寄与原因がソフトウェア内部にあり、RCM もソフトウェアで実装する必要がある

### 7.3 下位クラスへの分割検討(箇条 5.3.5 関連)

高クラスの RCM を実装する項目と、他の項目とを分離することで、非 RCM 項目の安全クラスを下げることを検討した結果を記録する。

| 項目 | 分離の実現性 | 分離後のクラス | 決定 |
|------|-----------|------------|------|
| ARCH-xxx | 可(プロセス分離) | B | 分離採用 |
| ARCH-yyy | 不可(共通データ) | C | クラス C のまま |

## 8. クラス変更の要件

以下の事由が発生した場合、クラスを再評価する。

- システムリスク解析の結果変更
- 新たなハザード識別
- 意図する使用目的の変更
- ハードウェア外部リスクコントロール手段の追加・変更
- 分離設計の変更
