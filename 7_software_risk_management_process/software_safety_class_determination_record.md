> **本プロジェクトの位置づけ(注記)**
>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。実機・ハードウェアは存在せず、PC 単体で動作する仮想化モデル(シミュレータ)を対象とする。
>
> 題材の **Therac-25** は AECL(Atomic Energy of Canada Limited)が 1982 年に発売した実在の放射線治療装置であり、1985-1987 年に米国・カナダで発生した 6 件の過剰照射事故(6 名重症または死亡)は医療機器ソフトウェア安全性の代表的事例である。
>
> 安全クラスの決定は、**仮に本ソフトウェアが実機の Therac-25 相当機に搭載されたと仮定した場合** の危害重大度に基づいて行う。本書は当時の AECL 社内文書の再現ではなく、公開された事故報告書・学術文献(Leveson & Turner 1993 ほか)に基づく学習用の再構成である。

## 1. 目的

本書は、IEC 62304 箇条 4.3 に基づき、Therac-25 Virtual Control Software(TH25-CTRL) のソフトウェア安全クラスを決定し、その根拠を記録する。本書は、ソフトウェア開発計画書(SDP)、ソフトウェア要求仕様書(SRS)、ソフトウェアリスクマネジメント計画書(SRMP) 等、後続のすべてのライフサイクル成果物の前提となる。

## 3. ソフトウェアシステムの概要

- **ソフトウェア名称:** Therac-25 Virtual Control Software(TH25-CTRL)
- **意図する使用目的:** 処方された放射線治療計画(線量、照射時間、ビームモード、エネルギー、コリメータ開口、ターゲット位置)に従い、電子ビーム(電子線治療モード)および制動放射線(X 線治療モード)の発生・制御・モニタリングを行うための治療制御ソフトウェア。本プロジェクトでは当該機能を PC 上のシミュレータで再現する。
- **動作環境(想定):** 本来は二重冗長 PDP-11 相当の治療制御コンピュータ上で動作する組込み環境。本プロジェクトでは PC 単体(Linux / macOS / Windows)で動作する仮想環境とし、ハードウェア I/O(電子銃、ベンディング磁石励磁電源、X 線ターゲット用ターンテーブル、コリメータ、光電式/機械式インターロック、ドーズモニタ(2 組のイオンチェンバ)、操作者コンソール)は C++ によるソフトウェアモデルで置換する。
- **ユーザ・操作者:** 医療従事者(放射線治療医、医学物理士、放射線治療技師)。患者は被操作対象であり操作者ではない。
- **主な機能:**
  - 治療モード(電子線 / X 線)とエネルギー(5-25 MeV 相当)の選択と確定
  - 電子ビーム生成・ベンディング磁石制御・ターンテーブル位置決め(電子線治療モード: 散乱箔、X 線治療モード: ターゲット + フラットニングフィルタ)
  - コリメータ開口・ジョー位置の制御
  - ドーズモニタ値に基づくビーム自動停止(閉ループ制御)
  - インターロック監視(ドア、ターンテーブル位置、ビーム電流、冷却水、真空度 等)
  - 操作者コンソール入力の受付・確定・ビームオン指令
  - 異常時のエラー報告・ロギング

## 4. ソフトウェア安全クラスの定義(IEC 62304 箇条 4.3 より)

| クラス | 定義 |
|--------|------|
| A | 傷害又は健康被害はあり得ない |
| B | 重傷には至らない傷害があり得る |
| C | 死亡又は重傷があり得る |

クラス決定は、**ハードウェア外部リスクコントロール手段を考慮しない前提** で、ソフトウェアが寄与し得るハザード状況から生じ得る危害の重大度に基づく。

本プロジェクトでは、Therac-25 の実事故(1985-1987、6 件、6 名重症/死亡)という **公知の決定的事例** が存在するため、クラス C 該当は事実上自明であるが、規格の要求事項に沿って根拠を明示的に文書化する。

## 5. ハザード寄与解析の要約

Therac-25 の公知事故事例(Leveson & Turner 1993)および放射線治療装置に関する一般的ハザード(IEC 60601-2-1)を参照し、本ソフトウェアが寄与し得るハザード状況を抽出した。

| ハザード ID | ハザード状況 | 想定される危害 | 重大度(システムレベル) | ソフトウェア寄与有無 | Therac-25 実事故との対応 |
|-----------|-----------|------------|---------------------|---------------------|---------------------|
| HZ-001 | **電子ビーム高出力モード** のまま X 線用ターゲット・フラットニングフィルタが挿入されない状態で患者に照射される(処方量の 100 倍規模の過剰被曝) | 急性放射線障害・皮膚潰瘍・組織壊死・死亡 | 致命的 | あり | **直接対応**(Tyler、Hamilton、East Texas 等、全事故の中核原因) |
| HZ-002 | 操作者入力処理と磁場/ターンテーブル制御間の **タイミング競合(race condition)** により、設定変更が未反映のままビーム照射が実行される | HZ-001 と同等 | 致命的 | あり | **直接対応**(Tyler 事故 1 回目: 8 秒以内の編集で発現) |
| HZ-003 | 共有フラグ変数の **周期的ゼロ値(整数オーバフロー)** により、本来行われるべき安全チェック(ターンテーブル in-position 確認)がバイパスされる | HZ-001 と同等 | 致命的 | あり | **直接対応**(Yakima 事故: Class3 変数のオーバフローバグ) |
| HZ-004 | **インターロック監視ロジック**(ドア、ビーム電流、ターンテーブル位置、ドーズモニタ)の検知失敗・実装不備により、危険状態が検知されずビームが継続 | 被曝継続による致命的障害 | 致命的 | あり | **直接対応**(ハードウェアインターロック廃止→ソフトウェア依存の設計判断) |
| HZ-005 | ビーム出力・ビームパルス数の計算/積算誤り(単位変換誤り、丸め誤差、オーバフロー)による過剰/過少被曝 | 過剰被曝による急性放射線障害・死亡、または過少被曝による治療失敗 | 致命的 | あり | 部分対応(Therac-25 では直接的原因ではなかったが、一般的放射線装置の既知リスク) |
| HZ-006 | 暗号的なエラーメッセージ(例: "MALFUNCTION 54")と **操作者によるバイパス常態化**(P キー押下で照射継続)の UI 設計 | 致命的な異常を異常と認識できず、照射が反復継続される | 致命的 | あり | **直接対応**(全事故で共通して観察された運用パターン) |
| HZ-007 | 旧機種(Therac-6/20)由来の **コード再利用における前提条件見直し漏れ**(旧機種ではハードウェアインターロックが同等機能を担っていた) | ソフトウェア単独では検知できない危険状態の潜伏 | 致命的 | あり | **直接対応**(AECL 事故後分析の主要指摘事項) |
| HZ-008 | ドーズモニタ(2 組のイオンチェンバ)の読み取り処理・比較ロジックの誤り(片系失陥時のフォールバック、サチュレーション検知等) | 過剰被曝を検知できずビーム継続 | 致命的 | あり | 部分対応(閉ループ制御の妥当性に関する一般的リスク) |
| HZ-009 | 電源喪失・再起動時の不定動作(治療中断状態からの誤った再開、ビームオン状態の引き継ぎ) | 意図しない二重被曝・過剰被曝 | 致命的 | あり | 部分対応 |
| HZ-010 | 操作者ログイン・権限管理の不備による、未承認の処方変更・ビーム設定変更 | 不適切な治療条件での照射 | 重大〜致命的 | あり | 潜在リスク(サイバーセキュリティ観点) |

## 6. 最悪シナリオにおけるソフトウェア起因の危害

ハードウェア外部リスクコントロール手段を仮に無しとした場合、ソフトウェア故障により生じ得る最悪シナリオを評価する。**いずれのシナリオも Therac-25 実事故で現実化している** ため、最悪シナリオ想定ではなく「実績ある最悪事例」である。

| シナリオ | 想定される危害 | 重大度 |
|---------|------------|-------|
| X 線モードで編集された治療計画が 8 秒以内に電子線モードに再編集された際、モード変更指令と磁場電流制御のタイミングが不整合となり、電子ビーム高出力(通常 X 線モードで使用)のまま X 線ターゲット非挿入で照射 | **死亡又は重傷**(Tyler 第 1 例: 推定 1.5 万 rad 相当被曝、患者死亡) | 死亡又は重傷 |
| 共有フラグ(例: Class3 変数)が 256 回の周期カウンタで 0 にリセットされる瞬間にターンテーブル位置チェックを通過し、ターンテーブルが in-position でない状態でビーム照射 | **死亡又は重傷**(Yakima 事故: 患者死亡) | 死亡又は重傷 |
| "MALFUNCTION 54" が表示されるもビームは停止せず「P」キー継続により反復照射、操作者が異常と認識できず 2 回目の致死的被曝 | **死亡又は重傷**(East Texas 事故) | 死亡又は重傷 |
| インターロックロジックのデッドロックにより、ドア開放・ビーム異常時にビームオフ指令が発行されない | 室内医療従事者・患者の被曝 | 死亡又は重傷 |

## 7. 安全クラスの決定

### 7.1 決定結果

**本ソフトウェアの安全クラス: C**

### 7.2 根拠

以下の観点から、本ソフトウェアはクラス C に該当する。**本判定は Therac-25 実事故により実証された危害であり、想定ではない**。

- [x] 故障時に死亡又は重傷が生じ得るハザード状況に寄与する(HZ-001〜HZ-010 のうち HZ-001〜HZ-009 が致命的、HZ-010 が重大〜致命的)
- [x] ハードウェア外部リスクコントロール手段(機械的インターロック、独立ハードウェアによるビームオフ等)が廃止/不十分な前提で、ソフトウェアがビーム制御・安全監視の一次担当である
- [x] 寄与原因がソフトウェア内部(タスク間同期、共有変数制御、状態遷移、UI 入力確定、ドーズ計算、インターロック監視)にあり、リスクコントロール手段(RCM)もソフトウェアで実装する必要がある
- [x] 公知事例(Therac-25 事故、1985-1987)において、ソフトウェア起因の死亡・重傷事例が **実際に** 6 件発生している
- [x] 放射線治療装置は IEC 60601-2-1(電子加速器型医用電気機器)の対象機器であり、過剰被曝は不可逆的危害である

### 7.3 下位クラスへの分割検討(箇条 5.3.5 関連)

高クラスの RCM を実装する項目と、他の項目とを分離することで、非 RCM 項目の安全クラスを下げることを検討する。本時点ではアーキテクチャ未確定のため、想定される分割候補とその方針を記録する。詳細な決定はアーキテクチャ設計書(箇条 5.3)にて行い、本書を改訂する。

| 項目(仮称) | 分離の実現性 | 分離後のクラス(想定) | 決定 |
|------|-----------|------------|------|
| ビーム制御コア(電子銃、磁場、ターンテーブル位置決め) | 分離不可(安全コア) | C | クラス C のまま |
| インターロック監視・ビームオフ指令発行 | 分離不可(安全機能) | C | クラス C のまま |
| ドーズモニタ処理・自動ビーム停止 | 分離不可(安全コア) | C | クラス C のまま |
| 治療計画データの入力確定・妥当性検証 | 分離不可(UI 層と安全コアの境界) | C | クラス C のまま |
| 治療履歴ログ表示(読み取り専用 UI) | 可(プロセス/モジュール分離により制御系と独立化可能) | B もしくは A | アーキテクチャ設計時に再評価 |
| 使用統計の外部送信機能 | 可(制御系と完全分離) | A | アーキテクチャ設計時に再評価 |
| 診断・保守用ログエクスポート | 可(読み取り専用、制御系から隔離) | A | アーキテクチャ設計時に再評価 |

> **Therac-25 事故の教訓:** 当時の AECL は、操作者コンソール UI と治療制御ロジックが **同一プロセス・同一共有メモリ空間** で動作していたため、UI 側のタイミングが制御側の状態遷移に影響を与える race condition が発生した。本プロジェクトでは箇条 5.3.5 の **分離(Separation)** 要求を明示的に設計に織り込み、UI 層と安全コアのデータフロー・制御フローを分離することを想定する。

## 8. クラス変更の要件

以下の事由が発生した場合、クラスを再評価する。

- システムリスク解析の結果変更
- 新たなハザード識別(公開された他の放射線治療装置事故事例の反映を含む)
- 意図する使用目的の変更(対象エネルギー帯の追加、特殊治療モードの追加、IMRT/VMAT 等の高度機能追加)
- ハードウェア外部リスクコントロール手段の追加・変更(例: 独立した機械式インターロックの追加、独立ハードウェアによる二次ドーズモニタの追加)
- 分離設計の変更(アーキテクチャ変更に伴う項目別クラスの再決定)
- 本プロジェクトの「仮想」前提の変更(仮に実機展開や他治療装置への応用を検討する場合)
