> **本プロジェクトの位置づけ(注記)**
>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。実機が存在しないため「市販後フィードバック」「規制当局への報告」等の一部プロセスは運用できないが、本書では実製品プロセスの形式をそのまま記述し、本プロジェクトで運用するサブセットを各章で明示する。
>
> 題材の **Therac-25** は、過去の事故において **問題の根本原因の追跡・水平展開調査・恒久的試験ケース化が不十分** であったことが事故拡大の一因と指摘されている(Leveson & Turner 1993)。本書はこの教訓を反映し、(a) クラス C 特有の **水平展開調査の必須化**、(b) **Therac-25 事故 5 主要因チェックリスト** との突合、(c) 修正試験ケースの **恒久化(永続的回帰試験への組込)** を運用ルールとして明文化する。
>

> 本書では問題報告を **PRB-NNNN**(Problem Report)と表記する。GitHub の Pull Request(PR)と混同しないための明示的な区別である。従来の IEC 62304 テンプレート記載にある「PR-NNNN」は、本プロジェクトでは一律 **PRB-NNNN** を指すものとする。

## 1. 目的と適用範囲

本書は、IEC 62304 箇条 9(ソフトウェア問題解決プロセス)に基づき、Therac-25 Virtual Control Software(TH25-CTRL) の開発中および(実製品展開時には)市販後に発見されたソフトウェア問題を、記録・分析・解決・検証・傾向分析する一連の手順を定義する。問題解決に伴う変更管理は SCMP(箇条 8)に従う。

**適用範囲:**

- C++ ソースコード・C++ ヘッダ・ドキュメント・試験資産(GoogleTest)・SOUP・ツール・環境定義に関する問題
- 開発中(試験・レビュー・静的解析・Sanitizer で発見)および保守段階(SMP と連携)の問題
- 安全性影響(ハザード寄与の可能性)を持つ問題は必ず SRMP(箇条 7)のリスク影響解析を経由する
- **並行処理関連の問題(race condition、デッドロック、メモリオーダリング違反等)** は、安全性影響の有無に関わらず必ず SRMP §6.1 の 3 層検証(TSan + モデル検査 + 境界値試験)を再実施する

## 3. 問題報告の作成(箇条 9.1)

### 3.1 トリガ

以下のいずれかが発生した場合、問題報告(PRB-NNNN)を作成する。

- 試験(ユニット UT / 結合 IT / システム ST)での不合格
- コードレビュー・設計レビュー・ドキュメントレビューでの指摘(軽微な typo 等は直接修正可だが、RCM 実装箇所の指摘は必ず起票)
- 静的解析(clang-tidy / cppcheck / Include-What-You-Use)での新規検出
- **動的解析の検出**: AddressSanitizer / UndefinedBehaviorSanitizer / **ThreadSanitizer** による検出(TSan 検出は **重大度 Critical 候補として即時 §4.3 のクラス C 追加調査を起動** する)
- ファジング(libFuzzer)での新規 crash / timeout 検出
- 網羅率(llvm-cov)が安全クリティカル部の 100% 要求を満たさない場合
- 脆弱性スキャン(Trivy / OSV-Scanner / Dependabot)での新規検出
- CI 失敗が既知の不具合を示唆する場合
- SOUP の新規脆弱性公開(GitHub Advisory / CVE / NVD / LLVM Security / GCC Security)
- コンパイラ・標準ライブラリ更新後の挙動変化(SCMP §4.3 の HZ-007 リスク評価で検出)
- 市販後の顧客・サービスからのフィードバック(実製品展開時)
- 監査・規制当局からの指摘(実製品展開時)
- 自己レビューで発見した潜在的不具合
- **Therac-25 事故 5 主要因チェック(§4.3.1)** で類似パターンが検出された場合

**本プロジェクトでの運用:** 上段のうち、市販後フィードバック・規制当局指摘は発生しない。他のトリガは本プロジェクトでも適用する。

### 3.2 起票手段

- GitHub Issue に **ラベル `problem-report`** を付けて起票する。
- Issue 番号を `PRB-{Issue 番号}` として問題報告 ID とする(例: Issue #42 → PRB-0042)。
- 問題報告テンプレートは `.github/ISSUE_TEMPLATE/problem_report.md` として配置済み(2026-04-21、コミット f8aa4d4)。テンプレートには Therac-25 事故要因対応欄(HZ-NNN との対応)が含まれる。

### 3.3 問題報告の最低記録項目

| 項目 | 内容 |
|------|------|
| 問題報告 ID | `PRB-NNNN`(= GitHub Issue 番号) |
| 報告者 / 報告日 | Issue 作成者 / 作成日 |
| 問題の概要 | 一文で要約 |
| 発見時のソフトウェアバージョン | Git コミット SHA-1 またはタグ |
| 発見時の環境 | OS / コンパイラ・バージョン(clang 17 等)/ ビルド構成(Debug / Release / Sanitizer 有効化)/ CI またはローカル |
| 再現手順 | 最小再現手順(コピー&ペースト可能なコマンド・入力) |
| 期待される動作 | 要求仕様(SRS-xxx)との対応を明記 |
| 実際の動作 | 観測内容 |
| 重大度(初期判定) | Critical / Major / Minor / Trivial(§3.4) |
| 安全性影響(初期判定) | あり(HZ-xxx 関連)/ 無し / 要調査 |
| **Therac-25 事故主要因対応** | 直接 / 部分 / なし(§4.3.1 のチェックリスト結果) |
| 関連資料 | ログ、スクリーンショット、試験 ID(UT-xxx / IT-xxx / ST-xxx)、関連 SRS、関連 RCM、Sanitizer 出力 |

### 3.4 重大度の判定基準

| 重大度 | 定義 | Therac-25 文脈の例 |
|-------|------|------------------|
| Critical | 安全性影響あり(HZ-001〜HZ-010 のいずれかに寄与)、または基本機能が使用不能、**または TSan/UBSan が検出した未定義動作** | 電子/X 線モード不整合、race condition によるビーム指令ずれ、整数オーバフローによる安全チェックバイパス、インターロック検知失敗、ドーズ計算桁落ち、起動不能 |
| Major | 安全性影響はないが、主要機能の動作不良でワークアラウンドが困難 | UI 表示誤り、操作ログ欠落、特定モード切替時の遅延 |
| Minor | 限定的な機能不良でワークアラウンドあり | 特定入力での表示崩れ、非安全機能のパフォーマンス低下 |
| Trivial | 誤字、コメント誤り、ドキュメント表現の軽微な不備 | Markdown 整形、コメントの typo |

**Critical は必ず即時 SRMP §7 の変更影響解析にかけ、RMF を更新する。**
**TSan が検出した並行処理不具合は、たとえ表面症状が軽微であっても初期判定 Critical とする(Therac-25 race condition 事故の教訓: 表面症状に騙されない)。**

## 4. 問題の調査(箇条 9.2)

### 4.1 調査項目(必須チェックリスト)

以下すべてを調査し、Issue 本体またはコメントに記録する。

- [ ] 問題の再現性(再現率、再現条件、非再現条件、Sanitizer 有効/無効での差異)
- [ ] 根本原因分類(SRMP §4.2 の潜在原因カテゴリと整合):
  - [ ] 要求誤り(SRS の欠陥)
  - [ ] 設計誤り(アーキテクチャ/詳細設計の欠陥)
  - [ ] 実装誤り(コードのバグ)
  - [ ] **並行処理・タイミング不具合(race condition、デッドロック、ABA 問題、メモリオーダリング違反)**
  - [ ] **整数オーバフロー・周期的ゼロ値**
  - [ ] **共有状態の競合的アクセス**
  - [ ] **インターロック・防御層の欠落**
  - [ ] エラー処理の曖昧化・バイパス可能 UI
  - [ ] **レガシーコード再利用時の前提喪失**
  - [ ] 試験誤り(テストケース側の欠陥)
  - [ ] SOUP 起因(依存パッケージの不具合・脆弱性、コンパイラバグ)
  - [ ] 環境依存(コンパイラ、OS、ロケール、スケジューラ)
  - [ ] ドキュメント誤り
- [ ] 関連するソフトウェア項目(ARCH-xxx / UNIT-xxx)
- [ ] 影響を受けるリリース・構成(過去のベースラインへの影響評価)
- [ ] **患者・使用者・第三者の安全への影響評価**(ハザード HZ-xxx との関連)
- [ ] 関連する RCM(RCM-xxx)が機能していたか(機能していなかった場合は RCM の不備として SRMP にフィードバック)

### 4.2 調査記録

Issue 本体の調査結果セクションに以下を記録する。

| 項目 | 内容 |
|------|------|
| 調査者 | k-abe |
| 調査期間 | `YYYY-MM-DD` 〜 `YYYY-MM-DD` |
| 再現結果 | 再現可否・再現率・最小再現ケース・Sanitizer 出力 |
| 根本原因分析 | 原因の特定(§4.1 の分類) |
| 影響範囲 | 影響する CI ID、過去リリースへの遡及影響 |
| 安全性影響 | あり(HZ-xxx 関連)/ なし |
| Therac-25 事故主要因対応 | 直接 / 部分 / なし(§4.3.1 のチェックリスト結果) |
| RMF 更新要否 | 要 / 不要(要の場合は RMF 更新エントリ ID を記載) |

### 4.3 クラス C 特有の追加調査

クラス C ソフトウェアでは、以下を追加で調査する。

- [ ] **水平展開調査**: 類似原因(同じ根本原因カテゴリ)による潜在問題がコードベース内に存在しないか — **Therac-25 事故では同型バグが複数事故で発現したことから、本プロジェクトでは Critical 案件で必須**
- [ ] 問題が RCM 実装部にある場合、分離設計(箇条 5.3.5)の前提が崩れていないか
- [ ] 試験網羅率(ユニット分岐網羅 ≥ 90% / 安全クリティカル部 100% / クラス C 項目は MC/DC 相当)の要求が満たされていたか
- [ ] **冗長 RCM 全段崩壊** の有無(SRMP §3.4 で 2 段以上の冗長化を必須化しているため、片方の RCM 失効が許容される範囲か検証)

#### 4.3.1 Therac-25 事故 5 主要因チェックリスト(必須適用)

すべての Critical 案件、および並行処理・整数演算・共有状態に関する Major 以上の案件で、以下のチェックを実施する。該当する場合は **その類型での水平展開調査を必須** とする。

| チェック項目 | 該当例 | 該当時の追加対応 |
|------------|--------|----------------|
| **A. 並行処理・タイミング競合**(Tyler 第 1 例型) | UI スレッドと制御スレッドの共有変数に対するメモリバリア無しのアクセス、`std::atomic` 不使用、ロック取得順序の不整合 | TSan 全 Pass の確認 + モデル検査(該当 RCM 対象なら)+ 影響範囲の他スレッド境界の同型走査 |
| **B. 整数オーバフロー・周期的ゼロ値**(Yakima Class3 型) | unsigned カウンタの周期的リセット、`uint8_t`/`uint16_t` のラップアラウンド、安全チェックの分岐条件がカウンタ値に依存 | 全 unsigned 整数型に対する境界値ユニット試験追加 + 同型コード(`+= 1`、`++` 操作の安全性)走査 |
| **C. 共有状態の競合的アクセス** | グローバル変数、静的変数、シングルトン、`std::shared_mutex` の誤用 | 共有状態の設計レビュー再実施 + 可能ならメッセージパッシング/イミュータブル化への置換検討 |
| **D. インターロック・防御層の欠落・バイパス**(全事故共通) | 単一防御層への依存、防御層の論理矛盾、安全チェックのスキップ条件 | 冗長化の追加 + 防御層間の独立性レビュー |
| **E. 暗号的エラー・バイパス可能 UI**("MALFUNCTION 54" 型) | エラーコードのみの表示、操作者の継続操作で復旧、ログ記録なし | エラーメッセージの人間可読化 + バイパス可能エラーの限定列挙 + ログ強化 |
| **F. レガシーコード再利用時の前提喪失**(Therac-6/20 型) | コンパイラ更新後の挙動変化、ライブラリ更新後の API セマンティクス変化、過去仕様に依存した実装 | 依存前提の文書化 + バージョンロック + リグレッション試験追加 |

このチェックリストの結果(該当/非該当)を Issue コメントに残し、PRB の調査記録欄にもサマリを記載する。**1 つでも該当があれば、SRMP §4.2 の対応する潜在原因カテゴリの再評価を実施する**。

## 5. 関係者への通知(箇条 9.3)

安全性影響の評価結果に応じて、以下の関係者に通知する。本プロジェクトは単独開発かつ仮想のため、実製品プロセスを記述しつつ、本プロジェクトで運用する部分を明示する。

| 通知先 | トリガ | 通知手段 | 本プロジェクトでの運用 |
|-------|-------|---------|---------------------|
| 品質責任者 | 全問題 | GitHub Issue | k-abe(セルフ、Issue の状態遷移で記録) |
| CCB | 修正を要する問題 | Issue + PR(SCMP §4.1.1) | k-abe(セルフ、1 分インターバル + CI 検証、学習プロジェクト特例、CCB-TH25-001 §5.4 で正式化予定) |
| RA 責任者 | 安全性影響のある問題 | Issue に `regulatory-impact` ラベル | k-abe(セルフ、該当時に RMF 更新を必須化) |
| 顧客 | フィールド対応必要 | カスタマー通知 | 運用なし(仮想) |
| 規制当局 | 法定報告要件該当 | 薬機法報告、FDA MDR 等 | 運用なし(仮想)。実製品展開時に SMP で追加規定 |

**重大度 Critical の場合の処理フロー(本プロジェクト版):**

1. Issue に `severity-critical` + `problem-report` ラベル付与
2. SRMP §7 変更影響解析チェックリストを Issue コメントに貼り付けて実施
3. **§4.3.1 Therac-25 事故 5 主要因チェックリストを実施し、該当類型を明示**
4. RMF 更新が必要なら該当エントリ作成、Issue にリンク
5. 修正 CR を起票(SCMP §4.1 重大区分)

## 6. 変更管理プロセスの使用(箇条 9.4)

解決のための変更は、SCMP(箇条 8)の変更制御プロセスに従って実施する。

**フロー:**

```
[問題報告 PRB-NNNN 起票] → [調査・根本原因特定] → [変更要求 CR-NNNN 起票(SCMP §4.1.3)]
 → [SCMP §4.1 区分判定・承認] → [フィーチャブランチで実装] → [PR 作成] →
   [CI Pass + 自己レビュー] → [マージ] → [ベースライン/リリース反映] → [PRB クローズ]
```

**規約:**

- Issue(PRB)と PR は GitHub の `Fixes #NNNN` / `Closes #NNNN` 構文で相互リンク
- コミットメッセージに `Refs: PRB-NNNN, CR-NNNN` を含める
- PR 本文に影響範囲・関連 SRS / RCM / ハザードを記載(SCMP の PR テンプレート強制)
- リリースノートに含まれる PRB / CR を列挙
- **並行処理関連の修正 PR では TSan 全 Pass の証跡を PR 本文または CI ログへのリンクで明示**(SCMP §4.3 の重大かつ並行処理関連区分に準拠)

## 7. 記録の維持(箇条 9.5)

問題報告とその解決記録は、構成管理下に保管する。

**保管される情報:**

- 当初の問題報告(Issue 本体・コメント)
- 調査結果(Issue コメントでの記録、§4.3.1 Therac-25 事故主要因チェック結果含む)
- 関連する CR-NNNN と変更内容(PR の diff、マージコミット SHA)
- 検証結果(試験 ID、Sanitizer 出力、網羅率の変化)
- クローズ日、クローズ承認者

**保管方法:**

- GitHub Issue / PR は永続保管(削除禁止、SCMP §5.3 と整合)
- リリース時点のスナップショットは GitHub Releases + ベースラインタグで固定
- 保管期間: 実製品では医療機器の寿命 + 規定期間(SCMP と整合)。本プロジェクトは仮想のため学習アーカイブとして保持

## 8. 問題の傾向分析(箇条 9.6)

定期的に問題報告の傾向を分析し、プロセス改善・追加検証活動に反映する。

### 8.1 分析頻度

- **各インクリメント完了時(M1〜M4)** に必須
- リリース前(M_final)に総合分析
- 実製品展開時はマイナリリースごと(少なくとも年 1 回)

### 8.2 分析項目

- **発生件数:** モジュール別(ARCH-xxx 単位)、フェーズ別(要求 / 設計 / 実装 / 試験)
- **根本原因分類:** §4.1 のカテゴリで集計
- **修正リードタイム:** 起票からクローズまでの日数分布
- **再発:** 類似原因での再発件数(3 件以上で要因調査を起動)
- **安全性影響件数:** Critical の発生数、RMF 更新数
- **RCM 関連:** RCM 実装部で発生した問題の件数(プロセス是正のシグナル)
- **試験フェーズ逃れ率:** 後工程で発見された問題の、当初で捕捉すべきだった件数
- **Therac-25 事故主要因類型別件数(§4.3.1):** A〜F の各類型での発生件数を追跡。**3 件以上発生した類型については、対応する SRMP §4.2 の潜在原因カテゴリの再評価と RCM 強化を必須化**
- **Sanitizer 別検出件数:** ASan / UBSan / **TSan** 別の検出件数。TSan 検出は Therac-25 事故主要因 A 型に直結するため最優先で集計

## 9. ソフトウェア問題解決の検証(箇条 9.7)

すべての修正について、以下の観点で検証する。検証結果は PR 本文および対応する試験報告書に記録する。

- [ ] **問題が解消されたこと:** 再現試験が修正前に Fail し、修正後に Pass すること(**試験ケースを新規追加し恒久化** = 永続的回帰試験スイートに組込)
- [ ] **新たな不具合を導入していないこと:** 既存のユニット・結合・システム試験がすべて Pass(回帰試験)
- [ ] **リスクコントロール手段の有効性が維持されていること:** SRMP §7.4.2 のチェックリストを適用、該当 RCM の試験を再実行
- [ ] **構成管理・ドキュメントが整合していること:** SRS / SAD / SDD / CIL / RMF 等の関連ドキュメントが更新されていること
- [ ] **水平展開:** §4.3 のチェックを完了。とくに §4.3.1 の Therac-25 事故主要因チェックで該当した類型については、コードベース全体の同型箇所を走査済みであること
- [ ] **並行処理関連修正の場合**: TSan 全 Pass + (該当 RCM がモデル検査対象なら)モデル再検査 Pass(SRMP §6.1 / SCMP §4.3 と整合)
- [ ] **Sanitizer 関連修正の場合**: 該当 Sanitizer での再現が修正前 Fail / 修正後 Pass、CI に Sanitizer 試験ジョブが組込まれている

検証が合格した後、問題報告を「解決済」としてクローズする。GitHub では Issue を `closed` 状態にし、PR をマージする。

## 10. 試験文書の内容(箇条 9.8)

問題解決に伴う試験文書(試験計画書・報告書の追補、または恒久化された新規試験ケース)には以下を含める。

1. 試験対象の問題報告 ID(PRB-NNNN)/ 変更要求 ID(CR-NNNN)
2. 対象ソフトウェアバージョン(修正前 Git SHA・修正後 Git SHA / タグ)
3. 再現試験の手順と結果(修正前の Fail、修正後の Pass の両方を記録)
4. 回帰試験の範囲と結果(既存試験全 Pass を明示)
5. リスクコントロール手段の再検証結果(該当時 / 該当 RCM-xxx)
6. **Sanitizer 試験結果(ASan / UBSan / TSan、該当時)**
7. **モデル検査結果(TLA+ / SPIN、該当時)**
8. 試験環境とツールバージョン(OS、コンパイラ、ビルド構成、主要ライブラリ)
9. 実施日、実施者、承認者(本プロジェクトでは k-abe 単独、セルフ承認)
