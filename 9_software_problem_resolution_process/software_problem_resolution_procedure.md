## 1. 目的と適用範囲

本書は、IEC 62304 箇条 9(ソフトウェア問題解決プロセス)に基づき、開発中および市販後に発見されたソフトウェアの問題を記録・分析・解決・検証する一連の手順を定義する。

## 3. 問題報告の作成(箇条 9.1)

### 3.1 トリガ
以下のいずれかが発生した場合、問題報告(Problem Report; PRB-ID)を作成する。

- 試験(ユニット・結合・システム)での不合格
- コードレビュー・設計レビューでの指摘
- 市販後の顧客・サービスからのフィードバック
- SOUP の脆弱性・異常公開情報
- 監査・規制当局からの指摘

## 6. 変更管理プロセスの使用(箇条 9.4)

解決のための変更は、箇条 8 の構成管理プロセス(変更制御)に従って実施する。
- 問題報告 PRB-NNNN → 変更要求 CR-NNNN を起票
- CCB 承認 → 実装 → 検証 → マージ → リリース
- コミットメッセージ・Pull Request・リリースノートに `PRB-NNNN / CR-NNNN` を明記

## 7. 記録の維持(箇条 9.5)

問題報告とその解決記録は、以下の情報とともに構成管理下に保管する。

- 当初の問題報告(3 章)
- 調査結果(4 章)
- 関連する CR と変更内容
- 検証結果(試験 ID、カバレッジ影響)
- クローズ日、クローズ承認者

記録の保管期間: 医療機器の寿命 + 規定期間(SCMP と整合)

## 8. 問題の傾向分析(箇条 9.6)

定期的に問題報告の傾向を分析し、プロセス改善・追加検証活動に反映する。

### 8.1 分析頻度
- マイナリリースごと(少なくとも年 1 回)

### 8.2 分析項目
- モジュール別・フェーズ別の発生件数
- 根本原因分類(設計 / 実装 / 要求 / SOUP / 環境)
- 修正リードタイム
- 再発(類似原因)の有無
- 安全性影響のあった問題の件数・性質

### 8.3 フィードバック
分析結果に基づき、開発標準・試験計画・教育・ツール選定を見直す。

## 9. ソフトウェア問題解決の検証(箇条 9.7)

すべての修正について、以下の観点で検証する。

- [ ] 問題が解消されたこと(再現試験の不発生を確認)
- [ ] 新たな不具合を導入していないこと(回帰試験)
- [ ] リスクコントロール手段の有効性が維持されていること(7.4 再評価)
- [ ] 構成管理・ドキュメントが整合していること

検証が合格した後、問題報告を「解決済」としてクローズする。

## 10. 試験文書の内容(箇条 9.8)

問題解決に伴う試験文書には以下を含める。

1. 試験対象の問題報告 ID / CR ID
2. 対象ソフトウェアバージョン(修正前・修正後)
3. 再現試験の手順と結果(修正前の失敗、修正後の合格)
4. 回帰試験の範囲と結果
5. リスクコントロール手段の再検証結果(該当時)
6. 試験環境とツールバージョン
7. 実施日、実施者、承認者
