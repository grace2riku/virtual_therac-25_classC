# IEC 62304 開発ステップ記録(仮想 Therac-25 / 安全クラス C)

**本ドキュメントの位置づけ:**
本書は IEC 62304 に基づく医療機器ソフトウェア開発プロセスを、どの順序・どの根拠で進めたかを時系列で記録する。後続プロジェクトが IEC 62304 の進め方を参照する際のお手本となることを目的とする。

**更新ルール(リポジトリ運用規則):**
開発ステップが一つ完了するごとに、本ドキュメントに **ステップ番号・作業日・作業内容・成果物・採用根拠・関連コミット** を追記する。更新義務は CLAUDE.md に明文化されている。

## 全体戦略(フェーズ区分)

本プロジェクトは以下のフェーズで進める。各フェーズの意図を先に示すことで、個別ステップの採用根拠が俯瞰できる。

| フェーズ | 目的 | 主成果物 | IEC 62304 条項との対応 |
|---------|------|---------|-----------------------|
| **Phase 1: 分類と計画**(M0 開始期) | 安全クラス決定と開発プロセス全体計画の確立 | SSC、SDP | §4.3(分類)、§5.1(開発計画) |
| **Phase 2: 支援プロセス計画**(M0 基盤整備期) | リスク・構成管理・問題解決・保守の各プロセス計画を確立 | SRMP、SCMP、SPRP、SMP | §7、§8、§9、§6 |
| **Phase 3: インフラ整備**(M0 終了期) | ドキュメント公開・GitHub 運用・補助台帳の整備 | README、Issue/PR テンプレート、CIL、CCB 運用規程、CRR、RMF 初版 | §5.1.9、§5.1.10、§7.1.4 |
| **Phase 4: Inc.1〜Inc.4 実開発**(M1〜M4) | V字を機能インクリメントで一周ずつ実施。各 Inc. を Therac-25 事故主要因に対応 | SRS、SAD、SDD、C++ コード、UT/IT/ST 計画・報告 | §5.2〜§5.8 |
| **Phase 5: リリースと保守**(M_final 以降) | 全インクリメント統合、最終試験、v1.0.0 リリース、以降 SMP による保守 | SMS、リリース成果物、RMF 更新 | §5.8、§6 |

Phase 1〜2 の「計画先行」方針を採る根拠:

1. **IEC 62304 §4.3** は「安全クラスは他のすべてのプロセスに先行して決定する」ことを要求する。
2. **§5.1.1** は「開発プロセスを計画する」ことを要求し、SDP は他のすべての成果物から参照される上位計画となる。
3. 支援プロセス計画(SRMP/SCMP/SPRP/SMP)は、Inc.1 の SRS 作成以降で常時参照・運用されるため、**先に用意しないと後続工程が手戻りする**。
4. 先に計画書を完成させることで、Inc.1 以降の実開発時に「どのプロセスに従うか」の判断で悩まなくなる。

### Inc.1〜Inc.4 と Therac-25 事故要因の対応

各インクリメントの目的は、Therac-25 の事故要因に対応させ、**「この Inc. が予防した事故」「この Inc. で検証した仮説」** を明確にすることで、学習効果を最大化する。

| Inc. | 主要機能 | 対応する Therac-25 事故要因(SSC HZ-NNN) | 期待される学習成果 |
|------|---------|-------------------------------------|------------------|
| Inc.1 | ビーム生成・モード制御コア | HZ-001(電子/X 線モード不整合)、HZ-005(線量計算) | モード遷移の形式的定義、線量計算の境界値試験 |
| Inc.2 | インターロック/安全監視 | HZ-004(インターロック検知失敗)、HZ-008(ドーズモニタ処理) | 独立監視系の設計、冗長チェックの実装 |
| Inc.3 | タスク同期・タイミング保証 | HZ-002(race condition)、HZ-003(整数オーバフロー) | ThreadSanitizer でのレース検出、TLA+ によるモデル検査 |
| Inc.4 | 操作者 UI・エラー報告・ロギング | HZ-006(暗号的エラー)、HZ-010(権限管理) | IEC 62366-1 準拠、バイパス禁止 UI |

### Step 5 — SRMP(ソフトウェアリスクマネジメント計画書)作成

| 項目 | 内容 |
|------|------|
| 作業日 | 2026-04-24 |
| 作業内容 | SSC-TH25-001 v0.1 で確定済みの HZ-001〜HZ-010 を初期ハザードとして採用し、IEC 62304 §7 + ISO 14971:2019 に基づくリスクマネジメントプロセスを計画。Therac-25 事故主要因(race condition / 整数オーバフロー / インターロック検知失敗 / バイパス可能 UI / レガシーコード前提喪失)に対応する潜在原因カテゴリ・RCM 種別・検証手段を整備。並行処理 RCM については **TSan + モデル検査 + 境界値試験の 3 層検証を必須化**。SOUP 評価情報源を C++ エコシステム(LLVM Security、GCC Security、OSV、Trivy 等)に差し替え。事象シーケンス例として Tyler / Yakima / East Texas の 3 事故を因果連鎖で記述 |
| 成果物 | `7_software_risk_management_process/software_risk_management_plan.md`(SRMP-TH25-001 v0.1) |
| コミット | `5116a8e` — docs(7): ソフトウェアリスクマネジメント計画書を作成 (SRMP-TH25-001 v0.1, Step 5) |

**採用根拠:**

- **IEC 62304 §7.1**:ソフトウェアリスクマネジメントプロセスはアーキテクチャ設計(§5.3)・詳細設計(§5.4)で **常時参照** されるため、SRS / SAD 着手前に計画を確立する必要がある。
- 順序として **SSC → SDP → SRMP** とした理由:
  - SSC で安全クラスを確定 → SDP で開発全体プロセスを計画 → SRMP でリスク特定/制御の詳細プロセスを計画、という階層関係が IEC 62304 の構造に整合する。
  - SRMP は SDP §9(ソフトウェアリスクマネジメント計画)の詳細展開でもあり、SDP 後に作成するのが自然。
- **クラス C 特有要求への対応**: §7.1.4(潜在原因の文書化)、§7.2.2(ソフトウェア RCM の SRS 組込)、§7.3.3(トレーサビリティ)を本書 §4.4 / §5.2 / §6.3 で明示的に定義した。
- **本プロジェクト固有の追加要件(Therac-25 学習目的):**
  - クラス C 主要ハザードに **2 段以上の冗長 RCM** を必須化(§3.4)— Therac-25 当時の単一防御策崩壊への対応
  - 並行処理ハザードへの **3 層検証(TSan + モデル検査 + 境界値試験)** 必須化(§6.1)— Tyler 第 1 例 race condition への直接対応
  - ハザード識別を **トップダウン(SSC ハザードから)+ ボトムアップ(Therac-25 事故根本原因チェックリストから)** の 2 ルートで実施(§3.1)
  - 独立性担保時間間隔は SCMP-TH25-001(作成予定)で具体値を定義することとし、本書では暫定 1 分以上(姉妹プロジェクト CCB-VIP-001 §5.4 と同等の学習プロジェクト特例)と記載
- **SOUP の C++ エコシステム特性:** Python の `pip-audit` 単一手段と異なり、C++ では LLVM/GCC のコンパイラレベル既知バグ、libc++/libstdc++ 標準ライブラリの脆弱性、vcpkg/Conan 経由の依存ライブラリ、ヘッダオンリーライブラリ(GoogleTest 等)など多層的な情報源評価が必要なため、§4.3 で情報源を明示列挙した。

### Step 6 — SCMP(ソフトウェア構成管理計画書)作成

| 項目 | 内容 |
|------|------|
| 作業日 | 2026-04-24 |
| 作業内容 | IEC 62304 §8 に基づくソフトウェア構成管理プロセスを計画。GitHub Flow、CR 区分 3 段階(軽微/中程度/重大)、単独開発向け CCB プロセス、ベースラインタグ規則、SOUP 識別項目を C++ エコシステム(vcpkg/Conan、libc++/libstdc++、clang/gcc 等)に対応させて整備。Therac-25 固有の追加要求として、(a) 並行処理関連変更時の TSan 必須化、(b) コンパイラ・標準ライブラリ更新時の HZ-007(レガシーコード前提喪失)リスク評価を「重大変更の追加検証」として規定。事後記録(CRR クローズ・CIL バージョン昇格等)の admin bypass 例外条件(§4.1.2)を 4 条件で整備 |
| 成果物 | `8_software_configuration_management_process/software_configuration_management_plan.md`(SCMP-TH25-001 v0.1) |
| コミット | `f0122fe` — docs(8): ソフトウェア構成管理計画書を作成 (SCMP-TH25-001 v0.1, Step 6) |

**採用根拠:**

- **IEC 62304 §8 と SDP §11 の関係**: SDP §11(ソフトウェア構成管理計画)は SCMP への参照宣言である。SDP 完了後の早期に SCMP を整備することで、Inc.1 着手以前に Git ワークフロー・CR フロー・ベースライン規則が確定し、後続成果物が混乱なく構成管理下に置ける。
- 順序として **SRMP → SCMP** とした理由:
  - SCMP §4.3(変更検証)・§4.1.3(リスクマネジメント評価)が SRMP §7 を参照するため、SRMP を先に確定する必要がある。
  - 並行処理関連変更時の TSan 必須化は SRMP §6.1 の 3 層検証要求に基づくものであり、SRMP の枠組みを踏まえないと整合がとれない。
- **クラス C 特有要求への対応**:
  - 重大区分の判定に「RCM 実装部・クラス C 項目・SOUP 追加/更新・コンパイラ更新」を含め、安全クリティカルな変更が軽量プロセスを通り抜けないように設計
  - ベースラインタグ規則(`planning-baseline` / `incN-{requirements,design}-frozen` / `incN-baseline` / `vX.Y.Z`)で、設計凍結→実装→試験→リリースの各段階を不可逆に固定
  - リリースタグ・ベースラインタグは削除禁止(GitHub の保護タグ機能で強制)
- **Therac-25 学習目的の追加要件:**
  - **コンパイラ・標準ライブラリ更新を「重大変更」に明示分類**(§4.1)— Therac-6/20 から Therac-25 への移行時に「依存していたハードウェアインターロックの前提が消失」した教訓(HZ-007)を、現代コンテキストでは「コンパイラ・標準ライブラリ更新時の UB 挙動・メモリオーダリング・ABI 変化の前提見直し漏れ」と読み替えて反映
  - **並行処理関連変更時の TSan 必須化**(§4.3)— Tyler 第 1 例 race condition の直接対応として、CCB プロセス §4.1.1 のステップ 4(CI 機械検証)に明文化
