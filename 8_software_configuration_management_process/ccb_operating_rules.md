## 1. 目的と適用範囲

本規程は、IEC 62304 箇条 8.2(変更制御)に基づき、変更管理委員会(CCB: Change Control Board)の構成、運営、意思決定手順を定義する。本規程は {{製品名}} のソフトウェア・ドキュメント・SOUP・ツール・試験資産に対するすべての変更要求(CR)に適用する。

## 3. CCB の構成

### 3.1 常任メンバ
| 役割 | 所属 | 責任 |
|------|------|------|
| 議長 | 品質保証部門 | 会議運営、最終承認 |
| 書記 | 品質保証部門 | 議事録、CR 台帳更新 |
| 開発責任者 | ソフトウェア開発部門 | 技術的影響評価 |
| 試験責任者 | 検証部門 | 検証影響評価 |
| RA 責任者 | 規制対応部門 | 規制影響・報告義務評価 |
| リスクマネジメント責任者 | 品質保証部門 | 安全性・リスク評価 |

### 3.2 招集メンバ(必要に応じて)
- サービス部門
- カスタマーサポート部門
- マーケティング部門
- 外部専門家・コンサルタント

### 3.3 定足数
- **通常 CR**: 常任メンバの過半数かつ RA・RM 責任者のいずれかを含む
- **安全関連・回収関連 CR**: RA 責任者 + RM 責任者 + 品質保証責任者の全員参加

## 4. CR の起票と受付

### 4.1 起票可能な者
- 開発者・試験者
- カスタマーサポート・サービス
- 品質保証担当者
- RA 担当者
- 外部(顧客・規制当局)からの要求を受領した担当部門

### 4.2 起票時の記載項目
CR は課題追跡システム({{ツール名}})で管理し、以下を記載する。

- CR ID(自動採番)
- 起票者、起票日
- 変更対象(CI ID、ファイルパス)
- 変更理由(関連する PRB-ID、保守要求、規制要求)
- 変更概要
- 希望期限
- 初期重要度区分(軽微 / 中程度 / 重大)

## 5. CR の分類と承認フロー

### 5.1 区分定義

| 区分 | 定義 | 承認フロー |
|------|------|----------|
| 軽微 | コメント修正、誤字、ログ文言 | 開発責任者のみ |
| 中程度 | 非安全関連機能の修正・追加 | CCB 簡易審議 |
| 重大 | 安全関連機能、アーキテクチャ変更、SOUP 変更、リリースノート記載事項 | CCB 正式審議 |

### 5.2 分類判定の基準

以下のいずれかに該当する場合は **重大** と判定する:

- [ ] リスクコントロール手段(RCM)を実装する CI への変更
- [ ] ソフトウェア安全クラスに影響し得る変更
- [ ] 分離設計(箇条 5.3.5)に影響する変更
- [ ] 外部インタフェースの変更
- [ ] SOUP の追加・削除・バージョン変更
- [ ] 規制当局への通知義務が生じ得る変更
- [ ] 市販済み機器へのフィールド対応を要する変更

### 5.3 承認フロー図

```
[CR 起票]
    │
    ▼
[区分判定: 議長 + 関連責任者]
    │
    ├─ 軽微 → [開発責任者承認] → [実装] → [検証] → [クローズ]
    │
    ├─ 中程度 → [CCB 簡易審議] → [承認/却下] → ...
    │
    └─ 重大 → [CCB 正式審議] → [RM/RA 評価]
                                   │
                                   ├─ 承認 → [実装計画] → [実装] → [検証]
                                   │                                 → [CCB 完了確認] → [クローズ]
                                   └─ 却下/保留 → [起票者へ差戻し]
```

### 5.4 審議インターバルの運用

CR 起票から承認(または PR マージ)までに、即興的判断を避けるための **審議インターバル** を設ける。標準モデル(複数名 CCB)では定例会サイクルでこれが担保されるが、単独・小規模開発では CCB 複数名審議の「時間差による独立視点再検討」を擬制するため、意図的なインターバル設定を推奨する。

#### 5.4.1 プロジェクト規模別推奨値

| プロジェクト種別 | 推奨インターバル | 根拠・留意点 |
|---------------|--------------|----------|
| 学習・デモプロジェクト | 1 分以上 | 単独開発・仮想プロジェクトでの学習効率を優先(形式的な待機時間) |
| 小規模単独開発(実機未搭載) | 数時間 | 頭冷却時間と開発効率のバランス、同一セッション連続作業を避ける |
| 中規模チーム開発 | 12〜24 時間 | 複数名レビューの時間確保、タイムゾーン差の吸収 |
| 医療機器実機搭載プロジェクト | **24 時間以上**(標準) | 業界慣行値、安全クラス C における即興的判断の防止 |

派生プロジェクトは自プロジェクトの規模・目的に応じて値を選び、本セクションに選択値を明記する。運用中は Issue 起票時刻と PR 作成時刻の差分などで機械的に追跡可能とする。

#### 5.4.2 例外

- **軽微区分**(§5.1)はインターバル対象外
- **重大区分** でも、以下に該当する場合は CCB 議長判断でインターバル短縮可(ただし理由を PR 本文に明記):
  - 安全性影響のある重大不具合のホットフィックス(SMP 参照)
  - 規制当局指摘への緊急対応

## 7. 審議の観点

各 CR は以下の観点で審議する。

- [ ] 変更の必要性・妥当性(ビジネス価値 vs リスク)
- [ ] 技術的影響範囲(依存する CI、回帰試験範囲)
- [ ] 安全性への影響(RMF 更新の要否)
- [ ] 規制への影響(通知・申請の要否)
- [ ] スケジュール・コスト影響
- [ ] ロールバック計画の実施可能性
