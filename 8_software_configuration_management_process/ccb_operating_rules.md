> **本プロジェクトの位置づけ(注記)**
>
> 本ドキュメントは IEC 62304 に基づく医療機器ソフトウェア開発プロセスの学習・参考実装を目的とした **仮想プロジェクト** の成果物である。単独開発のため CCB は作成者が全役割を兼任する。本規程では、実製品運用の CCB 体制(複数名による委員会)を「標準モデル」として記述しつつ、本プロジェクトで実運用する「単独開発版」の代替手段を各節で明示する。
>
> 題材の **Therac-25** は、Leveson & Turner 1993 が事故主要因の一つとして「**単独開発者による安全クリティカルコード実装と独立レビューの不在**」を明示的に指摘している。CCB が果たす **複数名による独立レビュー機能** は、Therac-25 事故が示す通り安全クリティカルソフトウェアの致命的欠陥を未然防止する最後の砦である。本プロジェクトは単独開発のため CCB の独立性を擬制でしか実現できないが、その擬制手段(時間インターバル + 自己レビューチェックリスト + CI 機械検証)が **Therac-25 事故を防ぎ得たプロセス活動の追体験** となるよう設計する。

---

## 1. 目的と適用範囲

本規程は、IEC 62304 箇条 8.2(変更制御)および SCMP-TH25-001 §4.1 に基づき、変更管理委員会(CCB: Change Control Board)の構成・運営・意思決定手順を定義する。本規程は Therac-25 Virtual Control Software(TH25-CTRL)の C++ ソースコード・ドキュメント・SOUP・ツール・試験資産・CI 設定に対するすべての変更要求(CR)に適用する。

本規程は SCMP §4.1.1 の「単独開発向け CCB プロセス」を具体化し、独立した文書として監査時のトレーサビリティを確保することを目的とする。

## 3. CCB の構成

### 3.1 標準モデル(実製品運用想定)

| 役割 | 所属 | 責任 |
|------|------|------|
| 議長 | 品質保証部門 | 会議運営、最終承認、区分判定の裁定 |
| 書記 | 品質保証部門 | 議事録、CRR 更新 |
| 開発責任者 | ソフトウェア開発部門 | 技術的影響評価、実装可能性判断 |
| 試験責任者 | 検証部門 | 検証影響評価、回帰試験範囲決定、**Sanitizer 試験(ASan/UBSan/TSan)範囲決定** |
| RA 責任者 | 規制対応部門 | 規制影響・報告義務評価 |
| リスクマネジメント責任者 | 品質保証部門 | 安全性・リスク評価、RMF 更新判断、**Therac-25 事故 5 主要因チェック(SPRP §4.3.1)の実施判断** |
| 並行処理レビュア(ad hoc) | 開発部門 | 並行処理関連変更時の TSan 結果レビュー、モデル検査結果レビュー(Therac-25 race condition 対応の専門観点) |

### 3.2 招集メンバ(必要に応じて)

- サービス部門
- カスタマーサポート部門
- マーケティング部門
- 外部専門家・コンサルタント(放射線治療装置のセーフティエンジニアリング専門家、並行処理形式手法専門家等)

### 3.3 定足数(標準モデル)

- **通常 CR:** 常任メンバの過半数、かつ RA 責任者または RM 責任者のいずれかを含む
- **安全関連・回収関連 CR:** RA 責任者 + RM 責任者 + 品質保証責任者の全員参加
- **並行処理関連 CR:** 上記 + 並行処理レビュアの参加(Therac-25 race condition 直接対応として必須)
- **コンパイラ・標準ライブラリ更新 CR:** 上記 + 開発責任者の参加(HZ-007 リスク評価担当として必須)

### 3.4 本プロジェクトの兼任運用(単独開発版)

本プロジェクトでは k-abe が以下の全役割を兼任する:

| 標準モデルでの役割 | 本プロジェクトでの担当 | 独立性の代替手段 |
|-----------------|---------------------|----------------|
| 議長・書記 | k-abe | GitHub Issue + PR で記録(機械的監査証跡) |
| 開発責任者 | k-abe | コードは単独記述 |
| 試験責任者 | k-abe | CI による機械的試験 Pass 必須(GoogleTest + Sanitizer 全種) |
| RA 責任者 | k-abe | SMP §4.7 により本プロジェクトでは規制通知は運用なし |
| リスクマネジメント責任者 | k-abe | RMF 更新必須、SRMP §7.4 影響解析チェックリスト遵守、SPRP §4.3.1 Therac-25 事故 5 主要因チェック必須 |
| 並行処理レビュア | k-abe | TSan + モデル検査(該当時)の CI 全 Pass、SRMP §6.1 3 層検証要求遵守 |

> **Therac-25 事故との対比:** Therac-25 当時の AECL では、CCB 機能が事実上欠如または機能不全であり、初期事故後も同型バグの遡及調査が行われなかった(Leveson & Turner 1993)。本プロジェクトの兼任運用は **「単独開発でも CCB 機能を擬制で維持し、独立レビューを最低限担保する」** ことが学習目的の中核である。

## 4. CR の起票と受付

### 4.1 起票可能な者

- 開発者・試験者(= k-abe、本プロジェクトでは全員)
- (実製品展開時)カスタマーサポート・サービス、品質保証担当者、RA 担当者、外部要求を受領した担当部門

### 4.2 起票手段

本プロジェクトでは **GitHub Issue** を CR の起票手段とする。ラベル `change-request` を付与する。Issue 番号が CR 番号となる(`CR-NNNN` = Issue #NNNN)。Issue テンプレートは `.github/ISSUE_TEMPLATE/change_request.md`(コミット f8aa4d4 で配置済)を使用する。

### 4.3 起票時の記載項目

SCMP §4.1.3 および `.github/ISSUE_TEMPLATE/change_request.md` に従う。主要項目:

- CR ID(Issue 番号)
- 起票者(GitHub handle)、起票日
- 変更対象(CI ID、ファイルパス)
- 変更理由(関連 PRB-NNNN、保守要求、規格要求、SOUP 更新、コンパイラ更新、改善提案)
- 変更区分(軽微 / 中程度 / 重大)
- 影響範囲解析
- SRMP §7.2 リスク影響解析チェックリスト
- **SPRP §4.3.1 Therac-25 事故 5 主要因チェックリスト結果(MAJOR の場合は必須)**
- **並行処理影響の有無(該当時は TSan 再実施計画)**
- RMF 更新要否
- 実装見積もり・完了目標マイルストーン
- 検証計画
- リリース方針

## 5. CR の分類と承認フロー

### 5.1 区分定義(SCMP §4.1 との整合)

| 区分 | 定義 | 承認フロー(標準モデル) | 本プロジェクトでの運用 |
|------|------|----------------------|---------------------|
| 軽微 | 誤字、コメント、整形、リンク修正、CI 設定の非機能変更 | 開発責任者のみ | セルフ承認、即時 PR 可 |
| 中程度 | 非安全関連機能の追加・変更、非 RCM モジュールの変更、ドキュメント章追加 | CCB 簡易審議 | セルフ承認 + **1 分インターバル**(学習プロジェクト特例、本書 §5.4)+ 自己レビューチェックリスト + CI 全 Pass |
| 重大 | 安全関連(RCM 実装部・クラス C 項目)、アーキテクチャ変更、要求変更、SOUP 追加/更新、**コンパイラ・標準ライブラリ更新**、リリース影響 | CCB 正式審議 | セルフ承認 + 1 分インターバル + 自己レビューチェックリスト + **RMF 更新** + **SRMP §7.4 影響解析** + **SPRP §4.3.1 Therac-25 事故 5 主要因チェック** + CI 全 Pass(並行処理関連時は **TSan 全 Pass 必須**) |

### 5.2 重大判定の基準

以下のいずれかに該当する場合は **重大** と判定する(CR 区分判定用チェックリスト):

- [ ] リスクコントロール手段(RCM-xxx)を実装する CI への変更
- [ ] ソフトウェア安全クラス(クラス C 項目)に影響し得る変更
- [ ] 分離設計(IEC 62304 §5.3.5)に影響する変更 — **特に UI 層と安全コアの分離(Therac-25 race condition の主要因への直接対応)**
- [ ] 外部インタフェース(仮想ハードウェアモデル I/F 含む)の変更
- [ ] **並行処理(`std::atomic` / `std::mutex` / `std::thread` 関連、共有変数)の変更** — Therac-25 Tyler 事故型
- [ ] **整数演算・カウンタ・周期処理の変更** — Therac-25 Yakima 事故型
- [ ] **インターロック・防御層・安全チェックロジックの変更** — Therac-25 全事故共通
- [ ] **エラー処理・エラー表示・操作確定 UI の変更** — Therac-25 East Texas 事故型
- [ ] SOUP の追加・削除・バージョン変更(MAJOR 更新を含む)
- [ ] **コンパイラ・C++ 標準ライブラリ実装の変更** — HZ-007(レガシーコード前提喪失)リスク評価必須
- [ ] **Sanitizer ランタイム(ASan/UBSan/TSan)・モデル検査ツールの変更** — 検証信頼性に直結
- [ ] 規格要求・規制要求への対応を伴う変更
- [ ] リリース影響のある変更(SemVer の MINOR 以上に該当)
- [ ] (実製品展開時)規制当局への通知義務が生じ得る変更、市販済み機器へのフィールド対応

### 5.4 インターバルの意義と本プロジェクト適用値(正式定義)

**目的:** 起票直後の即興的判断を避け、頭を冷やしての再確認時間を確保する。CCB 複数名審議の「独立視点による再検討」を時間差で擬制する。**Therac-25 事故では「単独開発者の即興的判断」が事故要因の一つであった**ため、本プロジェクトでもインターバル設定は学習目的の中核要素である。

**本プロジェクトの正式適用値:**

| プロジェクト種別 | 適用インターバル | 根拠 |
|---|---|---|
| **本プロジェクト(学習目的・仮想 Therac-25)** | **1 分以上** | 単独開発・仮想プロジェクトのため、24 時間待機は学習効率と乖離。最低限の頭冷却時間として 1 分を **本書 v0.1 で正式定義**(姉妹プロジェクト CCB-VIP-001 §5.4 と同等の学習プロジェクト特例) |
| 実機プロジェクト(参考、本プロジェクト適用外) | 24 時間以上 | 即興的判断を防ぐ熟慮時間として業界慣行値。Therac-25 事故では即興判断が事故拡大の一因であり、安全クラス C ソフトウェアでは特に重要 |
| 中規模チーム開発(参考) | 12〜24 時間 | 複数名レビューの時間確保、タイムゾーン差の吸収 |

> **重要:** 後続プロジェクトが本テンプレートを実機適用する際は、**本節の値を 24 時間以上に戻すこと**。Therac-25 事故の文脈では、即興判断回避のための時間インターバルは安全担保の最終防衛線である。

**遵守方法(本プロジェクト):**

- Issue 起票時刻と PR 作成時刻の差分が **1 分以上** であること(GitHub 上で機械的に追跡可能、`gh issue view N --json createdAt` で起票時刻取得)
- インターバル中に自己レビューチェックリストの冒頭項目を確認してから PR 作成に進む
- **熟慮が必要な場合は自主的に延長可能**(複雑な規程改訂・アーキテクチャ変更・並行処理設計変更などは 1 分では拙速と感じた場合に任意延長し、延長理由を PR 本文に記載)
- とくに **並行処理関連変更・コンパイラ更新は 1 分を最低値とし、自主的に数時間以上の延長を推奨**(Therac-25 race condition / HZ-007 への対応として慎重判断が必要)

**例外:**

- 軽微区分は対象外(インターバル不要)
- 重大区分でも、以下に該当する場合はインターバル短縮可(ただし理由を PR に明記):
  - 安全性影響のある重大不具合のホットフィックス(SMP §5.4 参照)
  - 規制当局指摘への緊急対応(本プロジェクトでは運用なし)

**本書 v0.1 施行時点の運用開始:**

CCB-TH25-001 v0.1 は本プロジェクトの **最初の CCB 規程** であり、施行日(2026-04-24)以降に起票される全 CR に本規程が適用される。Phase 0〜2 までのプロジェクト立ち上げ活動(Step 0〜9)は本規程施行前のため、CR 台帳への遡及登録対象外(代わりに `DEVELOPMENT_STEPS.md` にステップ記録として残る)。

## 6. CCB 会議の運営(標準モデル)

### 6.1 定例会(標準モデル)

- 頻度: 週次 または 隔週
- 時刻: 組織ルールで決定
- 形式: 対面・ハイブリッド・リモートのいずれか

### 6.2 臨時会(標準モデル)

以下のトリガで議長が臨時招集する。

- 安全関連 CR の発生
- 市販後重大不具合の発覚
- 規制当局からの照会・指示
- リリース直前の重大判断
- **TSan/UBSan が CI で新規検出した場合**(Therac-25 race condition 直接対応)
- **Therac-25 事故主要因チェック(SPRP §4.3.1)で 3 件以上発生類型が検出された場合**(SRMP §4.2 再評価トリガ)

### 6.3 本プロジェクトでの「会議」代替

単独開発のため物理的会議は行わない。代替として:

- **審議の代替:** GitHub Issue 上で区分判定チェックリスト・影響解析・リスク評価・SPRP §4.3.1 チェックリスト結果を記入することをもって「審議」とする
- **議事録の代替:** Issue 本文・コメント・PR レビューコメント・PR マージコメントを議事録として扱う
- **議事進行の代替:** Issue テンプレートの項目順に記入することで議事進行を擬制
- **定足数の代替:** CI 全 Pass(並行処理関連時は TSan 全 Pass を含む)と 1 分インターバル完了をもって定足数成立と見なす

## 7. 審議の観点

各 CR は以下の観点で審議する(Issue 本文に記入する必須項目)。

- [ ] 変更の必要性・妥当性(目的と期待効果)
- [ ] 技術的影響範囲(依存する CI、回帰試験範囲)
- [ ] 安全性への影響(RMF 更新の要否、関連ハザード HZ-001〜HZ-010)
- [ ] 規制への影響(本プロジェクトでは主に SMP §4.7 のリリースノート記載事項の判定)
- [ ] スケジュール・マイルストーン影響
- [ ] ロールバック計画(変更失敗時の復旧手順)
- [ ] 関連する既存 PRB-NNNN との整合
- [ ] **SPRP §4.3.1 Therac-25 事故 5 主要因(A〜F)該当性**(MAJOR の場合は必須):
  - [ ] A. 並行処理・タイミング競合(Tyler 第 1 例型)
  - [ ] B. 整数オーバフロー・周期的ゼロ値(Yakima 型)
  - [ ] C. 共有状態の競合的アクセス
  - [ ] D. インターロック・防御層の欠落・バイパス(全事故共通型)
  - [ ] E. 暗号的エラー・バイパス可能 UI(East Texas 型)
  - [ ] F. レガシーコード再利用時の前提喪失(Therac-6/20 型)
- [ ] **並行処理関連変更の場合**: TSan / モデル検査の再実施計画
- [ ] **コンパイラ・標準ライブラリ更新の場合**: HZ-007 リスク評価結果

## 9. CCB の監査と継続的改善

- **四半期ごと**(本プロジェクトでは各 Inc. 完了時)に過去の CR を振り返り、以下を SPRP §8 傾向分析と併せて評価:
  - CR のインターバル遵守率(本プロジェクトでは 1 分以上、本書 §5.4)
  - CR 区分判定の妥当性(後から区分を変更した件数)
  - RMF 更新漏れの有無
  - CI 失敗に起因する CR の件数(プロセス改善シグナル)
  - **Therac-25 事故主要因類型(SPRP §4.3.1 A〜F)別の CR 件数**(3 件以上発生類型は SRMP §4.2 再評価必須)
  - **並行処理関連 CR の TSan 全 Pass 遵守率**(100% 必須)
  - **コンパイラ更新 CR の HZ-007 リスク評価実施率**(100% 必須)

- 評価結果に基づき本規程を改訂する。改訂は本規程自体を対象とした「重大」区分の CR として扱う(メタ運用)。
