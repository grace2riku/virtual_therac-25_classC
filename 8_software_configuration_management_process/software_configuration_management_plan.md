## 3. 構成識別(箇条 8.1)

### 3.1 構成アイテム識別手段の確立(箇条 8.1.1)

すべての構成アイテムは以下の識別情報をもつ。

- **一意の識別子(CI ID):** 種別プレフィクス + 連番(例: `CI-DOC-001`、`CI-SRC-012`、`CI-SOUP-003`、`CI-TEST-005`、`CI-TOOL-002`)
- **名称**
- **種別:** ドキュメント / C++ ソースコード / C++ ヘッダ / 試験資産(GoogleTest) / SOUP / ツール / 環境定義 / ビルド成果物
- **バージョン:** Git コミットハッシュ(SHA-1)、リリースタグ、またはドキュメント冒頭のバージョン番号
- **保管場所:** リポジトリ内パス、または外部保管の場合は URL・参照方法

識別の実現手段:

| 手段 | 内容 |
|------|------|
| バージョン管理システム | Git(GitHub ホスティング)。<https://github.com/grace2riku/virtual_therac-25_classC> |
| コミット識別 | SHA-1 コミットハッシュ |
| リリースタグ規則 | `v{MAJOR}.{MINOR}.{PATCH}`(セマンティックバージョニング) |
| ベースラインタグ規則 | Inc. 完了時: `inc{N}-baseline` / `inc{N}-requirements-frozen` / `inc{N}-design-frozen` 等の英名タグ。M0 完了時は `planning-baseline` |
| ドキュメント識別 | ファイル冒頭に **ドキュメント ID + バージョン** を明記。ドキュメント ID は種別プレフィクス(SDP/SRS/SAD/SDD/SRMP/SCMP/SSC/RMF/SPRP/SMP/CIL/CCB/CRR 等)+ 製品コード(TH25)+ 連番 |
| CI 一覧の一元管理 | [`configuration_item_list.md`](./configuration_item_list.md) にて全 CI を一覧管理 |

### 3.2 SOUP の識別(箇条 8.1.2)

各 SOUP について、以下の情報を CIL に記録する。**C++ 標準ライブラリ実装(libc++ / libstdc++ / Microsoft STL)、コンパイラ(clang++ / g++ / MSVC)、ビルドツール(CMake / Ninja)、パッケージマネージャ(vcpkg / Conan)、テストフレームワーク(GoogleTest / GoogleMock)、Sanitizer ランタイムも SOUP として扱う**。

| 項目 | 記録内容 |
|------|----------|
| SOUP ID | `SOUP-001` 等の連番 |
| 名称 | パッケージ名 / ライブラリ名 |
| 製造業者/供給元 | 開発元組織・コミュニティ・ベンダ |
| 一意な識別子 | パッケージマネージャ上の名前 + リリース(例: `vcpkg/gtest@1.14.0`、`conan/fmt/10.1.1`) |
| バージョン | セマンティックバージョン(固定バージョンで依存宣言) |
| ライセンス | SPDX 識別子(例: BSD-3-Clause、MIT、Apache-2.0、Boost Software License 1.0、GPL-3.0-with-exception) |
| チェックサム | リリース成果物の SHA-256(vcpkg.json / conanfile.txt のロックファイル、または `vcpkg install --x-feature=...` 後の `installed/vcpkg/info/*.list` から取得) |
| 入手場所 / URL | アップストリームリポジトリ URL、パッケージレジストリ URL |
| 入手日 | `YYYY-MM-DD` |
| 使用目的 | TH25-CTRL 内での利用箇所(モジュール名 / ARCH-xxx) |
| 安全クラス | 依存先の利用箇所に応じて A/B/C を割当て |
| 脆弱性情報源 | OSV(<https://osv.dev/>)、GitHub Advisory Database、CVE/NVD、LLVM Security、GCC Security、ライブラリ独自 SECURITY.md |

> **Therac-25 固有の SOUP リスク:** Leveson & Turner 1993 が指摘した「Therac-6/20 由来コードの再利用前提見直し漏れ」(SSC-TH25-001 HZ-007)に対応するため、SOUP 更新時およびコンパイラ更新時には **依存している前提条件(strict aliasing、UB、メモリオーダリング、ABI 互換性)の再評価** を SRMP §7.2 のチェックリストで実施する。

バージョン固定は以下の方法で実施する(Inc.1 着手時に最終確定):

- **vcpkg を採用する場合**: `vcpkg.json` のマニフェストモード + `vcpkg-configuration.json` でベースライン commit 固定 + `vcpkg/installed/vcpkg/info/*.list` のチェックサム保存
- **Conan を採用する場合**: `conanfile.py` または `conanfile.txt` + `conan.lock`(Conan 2.x のロックファイル機能)
- いずれの場合も、ロックファイルを CI として Git 管理する

### 3.3 システム構成ドキュメンテーションの識別(箇条 8.1.3)

ソフトウェアシステム構成を定義するドキュメントを識別し、構成管理下に置く。

| ドキュメント種別 | 管理方法 |
|----------------|----------|
| ソフトウェアマスタ仕様書(5.8) | リリースタグと紐付け。リリース時にスナップショットを固定 |
| ソフトウェアアーキテクチャ設計書(SAD) | Git 管理、ベースラインタグで凍結 |
| ソフトウェア詳細設計書(SDD) | Git 管理、ベースラインタグで凍結 |
| 構成アイテム一覧(CIL) | Git 管理、ベースライン時にスナップショットを `configuration_item_list.md` のセクションとして追記 |
| リリース成果物(実行可能モジュール、共有ライブラリ、試験成果物、Sanitizer 試験ログ) | GitHub Releases にアーカイブ。ファイルの SHA-256 をリリースノートに記載 |
| 環境定義 | `CMakeLists.txt`、`vcpkg.json`(または `conanfile.*`)、ロックファイル、`.github/workflows/*.yml`、`.clang-tidy`、`.clang-format` を Git 管理 |

> 本プロジェクトは仮想であり物理ハードウェア BOM は存在しないが、「**仮想ハードウェアモデル**(電子銃、ベンディング磁石、ターンテーブル、コリメータ、イオンチェンバ、インターロックスイッチ等の C++ クラス群)」のソースコードは CI として一括管理する。これらは Therac-25 当時の物理機構の代替であるため、SAD で「仮想ハードウェアアーキテクチャ」として明示的に分離設計する。

## 4. 変更制御(箇条 8.2)

### 4.1 変更要求の承認(箇条 8.2.1)

すべての構成アイテムへの変更は、承認された **変更要求(CR: Change Request)** に基づいて実施する。本プロジェクトでは CR を GitHub Issue(ラベル `change-request`)で管理し、対応する PR にリンクする。

| CR 区分 | 判定基準 | 承認プロセス |
|--------|---------|-------------|
| **軽微** | 誤字、コメント修正、整形、リンク修正、CI 設定の非機能変更 | 作成者(k-abe)のセルフ承認。PR + CI Pass のみ |
| **中程度** | 非安全関連機能の追加・変更、非 RCM モジュールの変更、ドキュメントの章追加 | CCB プロセス(§4.1.1)を経由。1 分以上の検討期間(学習プロジェクト特例、CCB-TH25-001 §5.4 で定義予定)を設け、自己レビューチェックリストを適用 |
| **重大** | 安全関連(RCM 実装部・クラス C 項目)、アーキテクチャ変更、要求変更、SOUP 追加/更新、コンパイラ更新、リリース影響のある変更 | CCB プロセス + リスク影響解析(SRMP §7)必須。RMF 更新を伴う。**並行処理関連変更時は ThreadSanitizer 全 Pass を必須条件に追加** |

#### 4.1.1 CCB プロセス(単独開発版)

単独開発のため CCB は作成者が兼任するが、以下の規律で独立性を擬制する:

1. **Issue 起票:** 変更内容・理由・影響想定を記述
2. **インターバル:** 中程度以上は起票から **1 分以上** 経過後に PR 作成開始(学習プロジェクト特例、CCB-TH25-001 §5.4 で具体化予定。即興的判断を防ぐ目的は実機プロジェクト 24 時間ルールと同じで、本プロジェクトは最低限の頭冷却時間として 1 分を設定。**実機適用時は 24 時間以上に戻すこと**。Therac-25 事故では「単独開発者の即興判断」が事故要因の一つであった点に留意)
3. **自己レビューチェックリスト:** PR テンプレート(`.github/pull_request_template.md`)に埋め込み、すべての項目を Pass
4. **CI 機械検証:** 全チェック緑必須(ドキュメント lint、コード lint、ユニット試験、結合試験、網羅率、Sanitizer 全 Pass、SOUP 脆弱性スキャン)。**並行処理関連変更時は TSan を必須項目に含める(SRMP §6.1 に準拠)**
5. **判定記録:** PR マージコメントに承認判定と根拠を記載

#### 4.1.2 事後記録の取り扱い(軽微な台帳更新)

既に承認・実装・検証・マージ済みの CR / PR に対する **事後記録のみ** の変更は、新規 CR の起票対象とせず、元 CR に付随する記録行為として扱う。対象例:

- 変更要求台帳(CRR)の状態遷移・クローズ日の記入
- 構成アイテム一覧(CIL)のバージョン昇格反映
- リリースノート・マイルストーン・ベースラインラベルの付与

これらの事後記録を通常の CR / PR フローで処理すると「CR クローズ用の CR」が必要となり循環が生じる。この循環を避けるため、次の条件を **すべて** 満たす場合に限り、`main` への直接 push(Branch Protection の admin bypass)を許容する。

1. 変更内容が、既に承認・検証済み CR の事後記録のみであること(仕様・設計・実装・検証手順の変更を含まない)
2. 事前にローカルで CI 相当の lint(`npx markdownlint-cli2 "**/*.md"`)を実行し 0 エラーを確認していること
3. コミットメッセージに元 CR ID と事後記録である旨を必ず記載すること(例: `chore(CR-0042): close CRR after merge`)
4. bypass 操作は GitHub audit log に記録される。**リリース前 CCB 監査** および **各ベースライン確定時** に bypass 回数・対象・操作者を必ずレビューし、濫用を防ぐこと

本ルートは軽微な事後記録のみを対象とし、要求事項・設計・実装・試験結果への変更には適用しない。適用時は CR 台帳の備考欄に「bypass 使用」と明記する。

#### 4.1.3 変更要求の記録項目

| 項目 | 内容 |
|------|------|
| CR ID | `CR-NNNN`(通し番号、GitHub Issue 番号を流用可) |
| 要求者・要求日 | Issue 作成者・作成日 |
| 変更対象 | 影響する CI ID(複数可) |
| 変更理由 | 問題報告 ID(PRB-NNNN)、保守要求、規格要求、改善提案、SOUP 更新、コンパイラ更新等 |
| 影響範囲解析結果 | 影響する SRS / ARCH / UNIT / 試験 |
| リスクマネジメント評価 | SRMP §7.2 のチェックリスト結果、RMF 追記有無、**並行処理影響の有無** |
| 区分 | 軽微 / 中程度 / 重大 |
| 承認者・承認日 | k-abe・PR マージ日 |
| 関連 PR | PR 番号 |

台帳は [`change_request_register.md`](./change_request_register.md) に集約する。

### 4.2 変更の実装(箇条 8.2.2)

承認後の変更実装は以下のルールに従う。

- **ブランチ戦略:** GitHub Flow。`main` から機能ブランチを切り、PR で統合
- **ブランチ名:** `feature/<短い説明>` / `fix/<問題ID>` / `docs/<対象>` / `chore/<対象>` 等
- **PR 必須:** `main` への直接コミットは禁止(GitHub ブランチ保護ルールで強制)。例外は §4.1.2 の事後記録のみ
- **コミットメッセージ:** Conventional Commits 形式。CR ID を本文に含める(例: `Refs: CR-0042`)
- **レビュー:** 単独開発のためセルフレビュー + CI 機械検証
- **CI 自動実行:**
  - **ドキュメント検証**: markdownlint-cli2 / lychee(`.github/workflows/docs-check.yml` で実装済)
  - **C++ ビルド**: CMake + Ninja(Inc.1 実装開始時に追加)
  - **静的解析**: clang-tidy / cppcheck / Include-What-You-Use(同上)
  - **ユニット試験**: GoogleTest(同上)
  - **動的解析**: AddressSanitizer / UndefinedBehaviorSanitizer / **ThreadSanitizer**(同上)
  - **網羅率**: llvm-cov(MC/DC 相当の分岐網羅、安全クリティカル 100%、それ以外 90% 以上)
  - **SOUP 脆弱性スキャン**: Trivy / OSV-Scanner / Dependabot

### 4.3 変更の検証(箇条 8.2.3)

変更は、その規模と安全性影響に応じた検証を経て受け入れる。

| 変更規模 | 最低限の検証 |
|---------|-------------|
| 軽微 | CI 全 Pass(lint、該当ユニット試験) |
| 中程度 | CI 全 Pass + 該当機能のユニット・結合試験 Pass + リグレッション試験 Pass + ASan/UBSan Pass |
| 重大 | 上記 + システム試験 Pass + リスク再評価(RMF 更新) + 関連ベースラインの再確認 |
| **重大かつ並行処理関連** | 上記 + **TSan 全 Pass + (該当 RCM がモデル検査対象なら)モデル再検査**(SRMP §6.1 に準拠) |
| **重大かつコンパイラ・標準ライブラリ更新** | 上記 + **全試験再実施 + Sanitizer 全種再実行 + HZ-007(レガシーコード前提喪失)リスク評価**(SRMP §7.3 に準拠) |

### 4.4 変更のトレーサビリティ(箇条 8.2.4)

以下の連鎖を維持する。

```
[問題報告 PRB-nnnn / 保守要求 / 改善提案] → [CR-nnnn] → [Issue/PR] →
   [コミット(SHA)] → [マージコミット] → [ベースラインタグ] → [リリースタグ]
```

具体的な手段:

- Issue 記述に関連 SRS / RCM / ハザードを明記(Issue テンプレートで強制)
- PR 本文に CR ID、関連 Issue、影響する CI、リスク評価結果を記載(PR テンプレートで強制)
- コミットメッセージに `Refs: CR-nnnn` を含める
- リリースノートに含まれる CR を列挙
- CI で整合性検査(CR ID と Issue の存在、PR タイトル形式)を自動化する(後続で追加)
- **C++ ソースコード内の RCM 実装箇所には `// RCM-xxx` コメントを付与**(SRMP §5.2 に準拠)。`grep "// RCM-" --include="*.cpp" --include="*.hpp"` で全件抽出可能とする

## 5. 構成状態記録(箇条 8.3)

各ベースライン・リリース時点の構成アイテム一覧・状態を記録し、いつでも参照可能とする。

### 5.1 ベースライン設定タイミング

以下のタイミングでベースラインを凍結する。

| タイミング | タグ例 |
|-----------|-------|
| 基盤ドキュメント承認時(M0 終了) | `planning-baseline` |
| 各 Inc. の要求凍結時 | `inc{N}-requirements-frozen` |
| 各 Inc. の設計凍結時 | `inc{N}-design-frozen` |
| 各 Inc. の試験完了時 | `inc{N}-baseline` |
| 各リリース時 | `v{MAJOR}.{MINOR}.{PATCH}` |

### 5.2 状態記録の内容

各ベースラインで以下を記録する。

| 項目 | 内容 |
|------|------|
| ベースライン ID | `BL-{YYYYMMDD}-{NNN}`(例: `BL-20260424-001`) |
| ベースライン日 | `YYYY-MM-DD` |
| Git タグ | 該当タグ名 |
| Git コミット | SHA-1(完全な 40 文字) |
| 含まれる CI とバージョン | CIL の該当時点スナップショット(CIL に追記セクションとして残す) |
| 関連 CR | ベースラインに含まれる CR ID 一覧 |
| 検証成果物 | 該当時点の試験報告書 ID、Sanitizer 試験ログ、網羅率レポート |
| 承認者・承認日 | k-abe・日付 |
| 備考 | リリース目的・変更概要 |

GitHub Releases 機能を活用し、タグごとにリリースノートとアーカイブを作成する。

### 5.3 保管

- **Git リポジトリ:** GitHub 上で永続保管。タグ・コミットは書き換え不可(force push は main で禁止)
- **ローカルバックアップ:** 定期的なミラーリング(必要に応じて外部ストレージへ)
- **保管期間:** 実製品では医療機器の寿命 + 規定期間のアーカイブが必要。本プロジェクトは仮想のため学習アーカイブとして保持(実製品展開時は保管計画を SMP に追記)
- **削除制限:** ベースラインタグ・リリースタグは削除禁止(ブランチ保護ルール + GitHub の保護されたタグ設定で強制)

## 8. ブランチ保護ルール(GitHub)

`main` ブランチに対して以下を設定する(`scripts/setup_branch_protection.sh` で適用、Inc.1 着手前にセットアップ予定):

- [ ] PR 必須(直接 push 禁止)
- [ ] CI チェック Pass 必須(ドキュメント検証 + C++ ビルド/試験/Sanitizer は Inc.1 以降追加)
- [ ] PR を最新の main に対して更新必須(stale な PR は再同期が必要)
- [ ] 管理者にもルール適用
- [ ] force push 禁止
- [ ] ブランチ削除禁止
- [ ] タグ保護: リリースタグ・ベースラインタグは削除/上書き禁止
- [ ] admin bypass 例外: §4.1.2 の事後記録のみ(audit log 監査で運用統制)
